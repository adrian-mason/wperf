# ADR-009: IO Attribution — Synthetic Edges

- **Status:** Accepted
- **Date:** 2026-03-17

## Problem

The OSDI'18 wPerf paper treats I/O devices (disk, network) as "pseudo-threads" — virtual nodes in the Wait-For Graph that represent non-schedulable entities. When a user thread blocks on a disk read (e.g., via `fdatasync`), the paper models this as two edges: User->Disk (I/O initiated) and Disk->User (I/O completed, thread woken). This bidirectional edge structure allows pseudo-threads to participate in Tarjan SCC analysis and potentially be identified as part of a Knot — which is essential for correctly diagnosing I/O-bound bottlenecks (paper Figure 4).

The original wPerf implementation used JProbes on specific kernel functions (`blk_done_softirq`, `net_rx_action`, `net_tx_action`) to detect I/O completions and attribute wakeups to the corresponding subsystem. However, the JProbe API was removed from the Linux kernel in version 4.15, making this approach unavailable.

An early design iteration incorrectly excluded pseudo-threads from Knot detection, arguing that disk interrupt completions do not go through the normal scheduler wakeup path. This was subsequently identified as a deviation from the paper and reversed.

## Options Considered

### Option A: Paper's JProbe approach (instrument specific kernel functions)

Directly port the paper's approach by replacing JProbes with kprobes on the same internal kernel functions (`blk_done_softirq`, `net_rx_action`, `net_tx_action`).

**Pros:** Most faithful to the paper's original implementation. Direct function-level attribution — when `blk_done_softirq` runs and wakes a thread, the causal chain from I/O completion to thread wakeup is unambiguous.

**Cons:** These are internal kernel functions, not part of the stable kernel ABI. They may be renamed, refactored, inlined by the compiler, or have their signatures changed across kernel versions. The design review identified this instability (citing examples where `futex_wait_queue_me` was refactored to `futex_wait_queue` in some 5.x versions). kprobe attachment may silently fail if the symbol doesn't exist, or worse, attach to a differently-behaved function with the same name. This approach recreates the same fragility problem that necessitated abandoning JProbes in the first place.

### Option B: Synthetic edges with pseudo-threads (tracepoint-based)

Use stable kernel tracepoints (`block_rq_issue` / `block_rq_complete` for block I/O; `softirq_entry` / `softirq_exit` for softirq context identification) to detect I/O operations and their completions. When a `sched_wakeup` occurs within a `BLOCK_SOFTIRQ` context (as determined by the softirq tracking), attribute the wakeup to the Disk pseudo-thread. Explicitly construct ("synthesize") a bidirectional edge pair: User->Disk when the I/O is issued, and Disk->User when the I/O completes and the thread is woken.

**Pros:** All tracepoints used are stable ABI — `sched_switch`, `sched_wakeup`, `block_rq_issue`, `block_rq_complete`, `softirq_entry`, and `softirq_exit` have been stable across all supported kernel versions (4.18+). No dependency on internal function names or signatures. Pseudo-thread nodes (Disk, Network) represent device/subsystem abstractions rather than kernel implementation details. The synthetic edges do not break Tarjan's algorithm — SCC detection only examines adjacency relationships, not edge provenance. Event pairing uses `(device, sector, TID)` three-tuple matching for `block_rq` events, providing unambiguous I/O correlation.

**Cons:** Requires explicit event pairing logic in userspace (matching `block_rq_issue` with `block_rq_complete`). The softirq context tracking adds BPF-side state (tracking whether the current CPU is inside a softirq handler). Synthetic edges are an engineering abstraction not directly described in the paper — the paper uses direct function instrumentation rather than context-based attribution.

## Analysis

The spec chain's treatment of pseudo-thread participation in Knot detection went through a notable correction cycle. The first review identified that Tarjan SCC alone is insufficient for Knot detection — the condensation DAG must be built and only Sink SCCs (out-degree zero) qualify as Knots. The reply accepted this correction but went further, incorrectly arguing that pseudo-threads should be excluded from Knot participation entirely because "disk interrupt returns are not normal thread scheduler processes" and "pseudo-threads can only exist as directed graph leaf absorption points (Sink Collectors), not in Knots."

The subsequent expert review directly refuted this exclusion by citing the OSDI'18 paper's Figure 4, where I/O pseudo-threads are formally included in Knot evaluation. The reply document acknowledged this error and reversed the decision: "The system must capture cross-boundary events (`block_rq_complete` etc.) and synthetically generate a wakeup edge from Disk_IO to the user thread. As long as these two edges (User->Disk: I/O initiation, Disk->User: interrupt return) have matching time stamps, they must legitimately form a strongly connected graph and enter Tarjan settlement."

The choice of tracepoints over kprobes was driven by the broader decision to abandon all kprobe-based instrumentation of internal kernel functions. The design review identified that kprobe targets like `futex_wait_queue_me`, `try_to_wake_up`, and the various softirq handlers are unstable across kernel versions — they can be renamed, refactored, or inlined. The final spec adopted a strict policy: all core probes use stable tracepoints only. The paper's original approach used JProbes on these internal functions, but JProbes were removed in kernel 4.15, and kprobes on the same functions inherit the same instability.

The softirq context attribution mechanism replaces the paper's per-handler JProbes with a more general approach: `softirq_entry`/`softirq_exit` tracepoints track which softirq vector is active on each CPU. When a `sched_wakeup` fires within `BLOCK_SOFTIRQ` context (vector 4), it is attributed to the Disk pseudo-thread. Within `NET_RX_SOFTIRQ` context (vector 3), it is attributed to the Network pseudo-thread. This is more robust than hardcoding handler function names and automatically adapts to future softirq subsystems.

## Decision

Adopt Option B: Synthetic edges with pseudo-threads, using stable tracepoints for I/O event detection and softirq context attribution. Pseudo-threads are first-class citizens in the Wait-For Graph and participate fully in Tarjan SCC analysis and Knot detection.

## Consequences

- **Correctness alignment:** Restores paper-faithful I/O attribution after the incorrect exclusion in the intermediate design iteration. Pseudo-threads can appear in Knots, enabling correct diagnosis of I/O-bound bottlenecks.
- **Kernel stability:** All probes use stable tracepoints available on 4.18+ kernels. No dependency on internal kernel function names or signatures. Future kernel versions cannot break the instrumentation.
- **Event pairing complexity:** The `block_rq_issue`/`block_rq_complete` pairing requires matching by `(device, sector, TID)` three-tuple in userspace, adding state management to the event correlation pipeline.
- **BPF state:** softirq context tracking requires a per-CPU BPF map recording the current softirq vector between `softirq_entry` and `softirq_exit`. This is lightweight but must be correctly maintained.
- **Synthetic edge weight:** The Disk->User edge weight represents the I/O service time (`block_rq_complete_ts - block_rq_issue_ts`). The exact semantic definition and handling of concurrent I/O to the same device remain deferred to the implementation phase (Phase 2b).
- **Tarjan invariant preservation:** Synthetic edges do not affect the mathematical properties of SCC detection — Tarjan's algorithm operates purely on adjacency structure and is agnostic to edge provenance. This was explicitly confirmed in the review chain.
