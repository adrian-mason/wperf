# ADR-013: Scheduler Probe Type ‚Äî tp_btf/raw_tp vs Tracepoint

- **Status:** Accepted
- **Date:** 2026-03-18

## Problem

The final design specification (section 2.1) declares all scheduler probes using standard tracepoints:

| Paper Probe | wPerf Probe |
|---|---|
| `sched_switch` | `tp/sched/sched_switch` |
| `sched_wakeup` | `tp/sched/sched_wakeup` |

This decision was inherited from early design documents without empirical analysis. The question is whether raw tracepoints (`raw_tp/`) or BTF-enabled tracing (`tp_btf/`) would be more appropriate for wPerf's scheduler event collection, given the project's need for thread-level Wait-For Graph construction.

## Options Considered

### Option A: Standard Tracepoints (`SEC("tp/sched/sched_switch")`)

Use `BPF_PROG_TYPE_TRACEPOINT` programs attached to the scheduler's trace event format structs.

**Pros:**
- Available since kernel 4.7 ‚Äî widest compatibility
- Strongly typed context: `struct trace_event_raw_sched_switch *` with named fields (`prev_pid`, `next_pid`, `prev_state`, etc.)
- ABI-stable: the trace event format struct is part of the kernel's tracing ABI and rarely changes
- Direct field access: `ctx->prev_pid` with no helper calls

**Cons:**
- **Missing `tgid` field**: The `trace_event_raw_sched_switch` format exposes only `pid` (kernel TID), `comm`, `prio`, and `prev_state`. It does NOT include `tgid` (process ID / thread group ID). wPerf requires tgid to group threads by process for the Wait-For Graph
- **No access to `task_struct`**: Cannot traverse pointers to read cgroup ID, task flags, or other task metadata beyond the pre-formatted format fields
- **Formatting overhead**: The kernel formats ALL trace event fields into the flat struct on every invocation, regardless of which fields the BPF program actually reads
- **No CO-RE for format struct changes**: If a trace event format adds or renames fields, CO-RE relocations do not apply ‚Äî the format struct is not a kernel type with BTF annotations

### Option B: BTF-Enabled Tracing with Raw Tracepoint Fallback (`SEC("tp_btf/sched_switch")` + `SEC("raw_tp/sched_switch")`)

Use `BPF_PROG_TYPE_TRACING` (tp_btf) as the primary probe type, with `BPF_PROG_TYPE_RAW_TRACEPOINT` (raw_tp) as the fallback. Both receive the original tracepoint `TP_PROTO` arguments ‚Äî raw pointers to kernel structs ‚Äî rather than pre-formatted format fields.

**Pros:**
- **Full `task_struct` access**: Both `prev` and `next` task pointers are available, enabling `BPF_CORE_READ(prev, tgid)`, `BPF_CORE_READ(prev, flags)`, cgroup traversal, and any other task metadata
- **Lower overhead**: Skips the trace event formatting step ‚Äî the BPF program reads only the fields it needs directly from kernel memory
- **CO-RE compatible**: All `task_struct` field accesses go through `BPF_CORE_READ()` with full CO-RE relocations, compatible with `core_fixes.bpf.h` versioning patterns
- **Industry standard for scheduler tracing**: Every scheduler-tracing tool in libbpf-tools (offcputime, cpudist, runqslower, runqlat) uses this pattern. Zero scheduler tools use standard `tp/sched/`
- **tp_btf zero-copy**: On 5.5+ kernels, `BPF_PROG_TYPE_TRACING` uses BPF trampolines for near-zero invocation overhead and allows direct pointer dereference without `BPF_CORE_READ()`

**Cons:**
- **raw_tp requires kernel 4.17+**: Excludes kernels 4.7-4.16 (not in wPerf's target range ‚Äî minimum is 4.18)
- **tp_btf requires kernel 5.5+ with BTF**: Falls back to raw_tp on older kernels
- **TP_PROTO instability**: The `sched_switch` TP_PROTO changed in kernel 5.18 from 3 to 4 arguments (added `prev_state`). The change was backward-compatible (appended, not inserted), and the `BPF_PROG` macro handles it gracefully by ignoring extra arguments
- **`BPF_CORE_READ()` overhead in raw_tp path**: Each field access requires one `bpf_probe_read_kernel()` call. Negligible per-field (~5ns) but more than direct struct access in standard tp

### Option C: Standard Tracepoint with Supplementary pid‚Üítgid Map

Use standard `tp/sched/sched_switch` but maintain a BPF hash map populated by a secondary kprobe or tracepoint that maps `pid ‚Üí tgid`.

**Pros:**
- Uses the stable tp/ ABI for the primary event path
- Gets tgid through a side channel

**Cons:**
- Adds complexity: requires a secondary probe point, map maintenance, and garbage collection for exited processes
- Race condition: the pid‚Üítgid map may not be populated for short-lived threads when their first sched_switch fires
- Double overhead: both the formatting overhead of standard tp AND the map lookup overhead
- No production precedent in libbpf-tools ‚Äî no tool uses this pattern for scheduler events

## Analysis

### The `tgid` Problem Is Decisive

wPerf builds a per-process thread-level Wait-For Graph. Every edge in the graph connects two threads, identified by `(tgid, tid)` pairs. The `sched_switch` standard tracepoint format provides only `prev_pid` and `next_pid` ‚Äî which are kernel TIDs (what user-space calls thread IDs), not process IDs. The format struct does not include `tgid` because the tracing subsystem's format is designed for `trace-cmd` / `perf script` human-readable output, where PID and comm are sufficient.

With raw_tp or tp_btf, the BPF program receives `struct task_struct *prev` and `struct task_struct *next` directly, enabling:
```c
u32 prev_tgid = BPF_CORE_READ(prev, tgid);
u32 next_tgid = BPF_CORE_READ(next, tgid);
```

This is not a minor convenience ‚Äî it is a fundamental data requirement. The supplementary-map workaround (Option C) introduces race conditions and complexity that every production tool avoids by using raw_tp/tp_btf.

### libbpf-tools Empirical Evidence

Survey of all scheduler-tracing tools in `/workspace/kernel/bcc/libbpf-tools/`:

| Tool | Primary | Fallback | Standard tp? |
|------|---------|----------|-------------|
| offcputime | `tp_btf/sched_switch` | `raw_tp/sched_switch` | No |
| cpudist | `tp_btf/sched_switch` | `raw_tp/sched_switch` | No |
| runqslower | `tp_btf/sched_wakeup`, `tp_btf/sched_switch` | `raw_tp/sched_wakeup`, `raw_tp/sched_switch` | No |
| runqlat | `tp_btf/sched_wakeup`, `tp_btf/sched_switch` | `raw_tp/sched_wakeup`, `raw_tp/sched_switch` | No |

Standard `tp/sched/` is used only for non-scheduler events (syscalls, block I/O, etc.) where the trace event format provides all needed fields and `task_struct` traversal is unnecessary.

### Dual-Mode Implementation Pattern

The libbpf-tools pattern (from offcputime.bpf.c, offcputime.c):

**BPF-side:** Define both program variants in the same `.bpf.c`:
```c
SEC("tp_btf/sched_switch")
int BPF_PROG(sched_switch_btf, bool preempt,
             struct task_struct *prev, struct task_struct *next) {
    return handle_switch(prev, next);
}

SEC("raw_tp/sched_switch")
int BPF_PROG(sched_switch_raw, bool preempt,
             struct task_struct *prev, struct task_struct *next) {
    return handle_switch(prev, next);
}
```

Both call the same `handle_switch()` implementation ‚Äî the CO-RE `BPF_CORE_READ()` calls work identically in both program types.

**User-side:** Detect and select at startup:
```c
if (probe_tp_btf("sched_switch")) {
    bpf_program__set_autoload(obj->progs.sched_switch_raw, false);
} else {
    bpf_program__set_autoload(obj->progs.sched_switch_btf, false);
}
```

`probe_tp_btf()` (trace_helpers.c:1223) checks whether `BPF_PROG_TYPE_TRACING` programs can attach to the specified tracepoint. This follows the same "probe, reconfigure, load" pattern established in ADR-002.

### sched_switch TP_PROTO Version Difference

**Kernel 4.18 through 5.17:**
```c
TP_PROTO(bool preempt, struct task_struct *prev, struct task_struct *next)
```

**Kernel 5.18+:**
```c
TP_PROTO(bool preempt, struct task_struct *prev, struct task_struct *next,
         unsigned int prev_state)
```

The 4th argument was deliberately appended (not inserted) to preserve backward compatibility with existing BPF programs. Programs declared with 3 parameters via `BPF_PROG(name, bool preempt, struct task_struct *prev, struct task_struct *next)` work on both old and new kernels ‚Äî the 4th argument is simply ignored.

For `prev_state` access on pre-5.18 kernels, `BPF_CORE_READ(prev, __state)` (or `state` via `core_fixes.bpf.h`) reads the field directly from the task_struct, independent of the TP_PROTO.

### sched_wakeup TP_PROTO (Stable)

```c
TP_PROTO(struct task_struct *p)
```

Stable across all kernel versions from 4.18 to current. The waker's identity is obtained from `bpf_get_current_pid_tgid()`.

### Original wPerf Reference

The archived wPerf (`/workspace/kernel/wPerf-origin/`) used jprobes (a now-removed kernel mechanism) on `__switch_to()` ‚Äî the architecture-level context switch function ‚Äî with direct `task_struct` pointer access (`prev_p->pid`, `prev_p->tgid`, `prev_p->state`). raw_tp provides the equivalent access level through eBPF.

### Kernel Version Compatibility

| Kernel range | Probe type available | Notes |
|---|---|---|
| 5.5+ with BTF | `tp_btf/sched_switch` | Best: typed args, direct dereference, BPF trampoline |
| 4.17 ‚Äî 5.4 | `raw_tp/sched_switch` | Good: raw args, `BPF_CORE_READ()` required |
| 4.7 ‚Äî 4.16 | `tp/sched/sched_switch` only | No tgid access without workaround; below wPerf minimum |

wPerf's minimum kernel target is 4.18 (best-effort) / 5.4 (recommended). raw_tp is available on all target kernels. tp_btf is available on the recommended minimum and above.

## Decision

**Option B: tp_btf with raw_tp fallback** is chosen for all scheduler probe points.

The probe point table in `final-design.md` section 2.1 should be updated:

| Paper Probe | wPerf Probe (Primary) | wPerf Probe (Fallback) | Category |
|---|---|---|---|
| `sched_switch` | `tp_btf/sched_switch` | `raw_tp/sched_switch` | Core (mandatory) |
| `sched_wakeup` | `tp_btf/sched_wakeup` | `raw_tp/sched_wakeup` | Core (mandatory) |
| `sched_wakeup_new` | `tp_btf/sched_wakeup_new` | `raw_tp/sched_wakeup_new` | Core (thread creation) |
| `sched_process_exit` | `tp_btf/sched_process_exit` | `raw_tp/sched_process_exit` | Core (cleanup) |

Non-scheduler probes (futex, block I/O, softirq) retain standard `tp/` because their trace event format structs provide all needed fields and do not require `task_struct` traversal.

The user-space startup sequence adds a `probe_tp_btf()` check to the existing feature probe matrix from ADR-002:

| Feature | Probe Method | Degradation |
|---|---|---|
| **tp_btf** | `probe_tp_btf("sched_switch")` | Fall back to `raw_tp/sched_switch`; disable tp_btf programs via `set_autoload(false)` |

## Consequences

**Architecture:**
- Both tp_btf and raw_tp program variants exist in the single `wperf.bpf.o` ELF, sharing the same `handle_switch()` / `handle_wakeup()` implementations via static inline functions
- The user-space feature probe matrix (ADR-002) gains one additional probe: `probe_tp_btf()`. This slots naturally into the existing "probe, reconfigure, load" pattern
- All `task_struct` field access uses `BPF_CORE_READ()` with `core_fixes.bpf.h` patterns, working identically in both program types. The tp_btf path could use direct dereference for cleaner code, but using `BPF_CORE_READ()` uniformly keeps the shared implementation simple

**Implementation:**
- Each scheduler tracepoint requires two `SEC()` program definitions (tp_btf + raw_tp) in the BPF source. These are thin wrappers around shared logic ‚Äî estimated 8-12 additional lines per probe point
- The `BPF_PROG` macro from `bpf_tracing.h` handles argument unpacking for both program types, providing a uniform function signature
- `prev_state` access on pre-5.18 kernels reads from `task_struct` directly via CO-RE, independent of the TP_PROTO argument change

**Risk:**
- Future TP_PROTO changes to scheduler tracepoints could break the raw_tp argument layout. This risk is mitigated by the fact that kernel developers deliberately preserve backward compatibility for raw_tp (as demonstrated by the sched_switch 3‚Üí4 argument change being an append, not an insert). The `BPF_PROG` macro's argument handling provides additional resilience
- tp_btf availability depends on both kernel version (5.5+) and BTF availability. On kernels with BTF disabled (rare on 5.5+ but possible), the raw_tp fallback activates transparently
