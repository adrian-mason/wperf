# ADR-008: SCC Weight Heuristic

- **Status:** Accepted
- **Date:** 2026-03-17

## Problem

After the Cascade Engine redistributes weights and Tarjan's algorithm identifies strongly connected components (SCCs), the system must "condense" each SCC into a single super-node in the condensation DAG for critical path analysis. This requires assigning a single representative weight to each super-node. The choice of weight aggregation function directly affects bottleneck ranking — determining which SCCs (Knots) are presented to the user as the most severe performance problems. The question is: when an SCC contains multiple internal edges with different attributed weights (e.g., A->B at 100ms, B->C at 50ms, C->A at 30ms), what should the super-node's weight be?

## Options Considered

### Option A: MAX as mathematical necessity (concurrent blocking)

Take the maximum edge weight within the SCC. Justify this as a mathematical consequence of the "water bucket" principle: all threads in the SCC are concurrently blocked, so the overall bottleneck duration equals the longest single wait, analogous to the shortest stave determining the water level.

**Pros:** Simple to implement. Produces a conservative (lower-bound) estimate that never overstates the SCC's impact.

**Cons:** The "concurrent blocking" premise was proven to be mathematically unsound during review. The argument assumes all threads in the SCC are simultaneously in a blocked state during the same time window. In practice, large weight disparities (e.g., 500ms / 10ms / 10ms) indicate that the waits occur in different time windows — the blocking relationship is a sequential causal chain, not a concurrent deadlock-like state. If the waits were truly concurrent (as in a real deadlock), the weights would be approximately equal. The "water bucket" analogy does not hold because the threads are not blocking at the same time.

### Option B: SUM of all edges

Sum all internal edge weights to represent the total blocking cost within the SCC.

**Pros:** Captures the full accumulated waiting time within the cycle.

**Cons:** When threads do block concurrently, SUM double-counts overlapping time intervals. If A waits 100ms for B while B simultaneously waits 50ms for C, summing to 150ms creates 50ms of phantom time that never existed on any wall clock. SUM systematically overstates SCC severity and can dwarf legitimate non-SCC bottlenecks in the ranking.

### Option C: MAX as sorting heuristic (acknowledged approximation)

Take the maximum edge weight within the SCC, but explicitly document this as a sorting heuristic for bottleneck ranking rather than a mathematically precise attribution value. Acknowledge that MAX may underestimate in sequential-chain SCCs and that the "concurrent blocking" justification is incorrect.

**Pros:** Same simplicity and conservative property as Option A. Preserves relative ordering — an SCC with a 500ms max edge is almost certainly more impactful than one with a 50ms max edge. Honest about its limitations, preventing downstream consumers from treating it as an exact measurement.

**Cons:** Does not provide an exact attribution value. In pathological cases with highly asymmetric sequential chains, MAX may significantly underestimate the actual total blocking cost. No single scalar can perfectly represent a multi-edge cycle's impact without additional temporal analysis.

## Analysis

The initial proposal presented MAX as a mathematical certainty, using a "water bucket" analogy: "In the same causal cycle, these threads are concurrently blocking and waiting. The shortest stave of the bucket depends only on the longest thread (100ms)." This was directly challenged in the multi-expert review, which demonstrated the argument's flaw: an SCC with edges weighted 500ms, 10ms, and 10ms almost certainly represents sequential causal blocking across different time windows, not concurrent deadlock. If the threads were truly concurrently blocked, their wait times would converge to similar values.

The expert review provided a clear comparative analysis:

| Strategy | Semantics | Problem |
|----------|-----------|---------|
| MAX | Slowest single-point wait in the cycle | Underestimates total cost when weights differ significantly |
| SUM | Total wait cost of the cycle | Double-counts concurrent blocking periods |

The review's conclusion was precise: "Max is an acceptable approximation in Knot **ranking** scenarios (preserves relative magnitudes), but may underestimate in **absolute delay attribution** scenarios. The 'water bucket shortest stave' analogy is mathematically untenable." The recommendation was to explicitly document MAX as a sorting heuristic rather than a precise attribution value.

The final implementation spec adopted this framing verbatim, including a formal "serious declaration" that MAX is "merely a sorting heuristic for finding the system's most costly component, and does not represent a mathematically precise value of all threads simultaneously waiting equal amounts."

## Decision

Adopt Option C: MAX as a sorting heuristic with explicit acknowledgment that it is an approximation. The super-node weight in the condensation DAG equals `MAX(attributed_delay of all edges within the SCC)`.

## Consequences

- **Ranking correctness:** Relative ordering of Knots by severity is preserved in the vast majority of practical cases — a Knot with a 500ms maximum internal edge almost always represents a more severe bottleneck than one with a 50ms maximum.
- **Attribution limitation:** The MAX value should not be presented to users as "total time lost to this bottleneck." It is a ranking signal, not an absolute measurement. UI displays should label it accordingly (e.g., "max internal wait" rather than "bottleneck cost").
- **Implementation simplicity:** Single `max()` call over the edge set within each SCC. No additional temporal analysis needed at this stage.
- **Future improvement path:** A more accurate heuristic could analyze the temporal overlap of internal edges (using the same sweep-line machinery from the Cascade Engine) to determine whether the SCC's blocking is concurrent or sequential, and adjust the weight accordingly. This is deferred as the current heuristic is sufficient for bottleneck ranking.
- **Documentation obligation:** The "sorting heuristic" characterization must be preserved in code comments, API documentation, and user-facing reports. Future maintainers must not reintroduce the "mathematical necessity" framing.
