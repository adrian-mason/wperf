# ADR-015: I-1 Conservation Invariant — Per-Entry-Edge, Not Global Sum

- **Status:** Accepted
- **Date:** 2026-03-30
- **Supersedes:** The global sum conservation definition in the original §3.4 I-1 invariant table and ADR-012's weight conservation characterization.

## Problem

The original I-1 invariant was defined as: `Σ(attributed_delay) == Σ(raw_wait)` across all edges in the Wait-For Graph. During Phase 0 implementation (PR #59), this global sum equality was found to be mathematically incompatible with the per-edge independent cascade algorithm.

### Evidence

**Figure 4 proof (the paper's reference scenario):**

| Edge | raw_wait | attributed_delay |
|------|----------|-----------------|
| User→Parser | 100ms | 20ms |
| Parser→Network | 80ms | 80ms |
| **Σ** | **180ms** | **100ms** |

Global sum: 180 ≠ 100. This is not a bug — it is an inherent property of per-edge independent cascade.

**Why global sum conservation cannot hold:** Per-edge independent cascade computes `attributed = raw - propagated_downstream` for each edge independently. The propagated weight is subtracted from the entry edge's attributed delay, but it is NOT added to downstream edges — each downstream edge has its own independent cascade computation starting from its own `raw_wait`. Therefore weight is "absorbed" at intermediate nodes, and `Σ attributed < Σ raw` is a mathematical necessity.

**Why forcing global sum equality breaks I-2:** During Phase 0 implementation, an attempt to inject synthetic weight (`add_synthetic_attributed_weight`) to force `Σ attributed == Σ raw` caused I-2 (non-amplification) violations: Parser→Network's attributed became 160ms > raw 80ms.

### Origin of the original I-1 definition

The OSDI'18 paper (Zhou et al.) does NOT define any conservation invariant. The paper describes cascade as a recursive weight redistribution process and explicitly states that overlapping wait segments are "counted multiple times" (p.8, lines 640-648). The global sum conservation property was introduced as a project-specific verification invariant during our design phase (cascade-understanding.md §2.3, final-design.md §3.4) and was incorrectly assumed to be a property of the algorithm.

## Decision

Redefine I-1 as **per-entry-edge conservation**:

> **I-1 (Per-entry-edge conservation):** For each entry edge `e` in the Wait-For Graph, the sum of `attributed_delay` values across `e`'s cascade subtree equals `e.raw_wait`.

### Formal definition

An **entry edge** is any edge `(src, dst)` in the WFG. Its **cascade subtree** is the set of edges visited during `compute_cascade(dst, e.time_window, ...)` — i.e., the outgoing edges of `dst` that overlap `e.time_window`, their children's overlapping edges, and so on recursively.

Per-entry-edge conservation states:
```
∀ entry edge e:
  e.attributed_delay + Σ(child.attributed_delay for child in cascade_subtree(e)) == e.raw_wait
```

For Figure 4:
- Entry edge User→Parser: attributed=20, cascade subtree={Parser→Network: attributed=80}. 20 + 80 = 100 = raw_wait. ✓
- Entry edge Parser→Network: attributed=80, cascade subtree=∅ (leaf). 80 = 80 = raw_wait. ✓

This property:
1. Is **independent from I-2/I-3** — neither implies nor is implied by per-edge non-amplification or non-negativity
2. **Catches the same 4/5 bugs** as the original I-1 (BUG-2, BUG-3, BUG-4, NEW-BUG-1)
3. Is **executable and testable** — can be verified after each cascade run
4. **Does not require synthetic weight injection** — holds naturally when `compute_cascade` correctly uses the recursive return value per ADR-007 pseudocode

### What does NOT change

- I-2 through I-7 are unchanged
- `is_conserved` remains the production sentinel flag (but now checks per-entry-edge conservation)
- The verification pyramid structure (ADR-007) is unchanged
- The five-layer testing strategy is unchanged

**Implementation updates required (separate PRs):**
- PR-A: Fix `compute_cascade` to use recursive return value per ADR-007 pseudocode; fix Python oracle; add per-entry-edge conservation tests and cycle/depth-truncation regression tests
- PR-B: Update `is_conserved` implementation to check per-entry-edge conservation; `cascade_engine` returns `Result`; release mode enforcement

**Gate 0 / Phase 0 boundary clarification:**
- Gate 0 validates understanding of the paper's cascade behavior (Figure 4 trace, per-entry-edge conservation on simple graphs)
- Phase 0 owns the formalization and enforcement of project-specific invariants (I-1 through I-7 as executable checks, property-based verification across 10K+ graphs)

**Risk:** The amended I-1 is a weaker property than the original global sum equality. However, the original property was impossible to satisfy, making it vacuously strong. The amended property is the strongest correct conservation invariant for per-edge independent cascade.
