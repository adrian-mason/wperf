# ADR-005: Stack Unwinding Approach

- **Status:** Accepted
- **Date:** 2026-03-17

## Problem

wPerf-rs must capture accurate user-space call stacks during `sched_switch` events to attribute off-CPU wait times to specific code paths. Industrial C/C++/Rust binaries are compiled with `-fomit-frame-pointer` by default (GCC 4.6+, Clang), which means the RBP register is used as a general-purpose register and does not form a frame pointer chain. The naive `bpf_get_stackid` helper only returns meaningful multi-frame stacks when frame pointers are present. Without a reliable unwinding strategy, the tool cannot provide the call-stack context needed for actionable performance diagnosis.

## Options Considered

### Option A: Kernel-side DWARF via bpf_loop / pragma unroll

Parse `.eh_frame` DWARF CFI rules in userspace, load them into BPF hash maps keyed by PC ranges, and perform a 20-iteration unrolled loop in the BPF program (`#pragma unroll(20)` or `bpf_loop`) to walk the stack by reading user memory with `bpf_probe_read_user`.

**Pros:** Keeps all unwinding in kernel space; no raw stack memory shipped to userspace; theoretically 100% accurate.

**Cons:** `bpf_loop` was introduced in kernel 5.17 and is unavailable on 4.18. A 20-layer unroll produces 800-1200 instructions, likely exceeding the 4,096-instruction limit on 4.18 kernels (raised to 1M only in 5.2+). The BPF verifier on 4.18 may reject the program due to state space explosion. This approach was deemed fundamentally incompatible with the minimum supported kernel target.

### Option B: Userspace DWARF (BPF sends 128-512B raw stack)

Move DWARF parsing entirely to userspace. The BPF probe captures the current RSP, RBP, RIP, and a chunk of raw stack memory (128-512 bytes) as event payload, ships it via ringbuf to userspace, where Rust code performs the full DWARF CFA walk using the `gimli` crate.

**Pros:** Completely bypasses BPF instruction limits and verifier complexity; standard DWARF handling in Rust.

**Cons:** At 100K events/s, shipping 512 bytes per event produces ~50 MB/s of ringbuf traffic, threatening buffer overflow. User-space pages may be swapped out during `sched_switch`, causing `bpf_probe_read_user` to return `-EFAULT`. The bandwidth explosion problem (designated N-1/N-3 in review) was identified as a showstopper for high-frequency workloads.

### Option C: sys_enter pre-store + RBP three-frame hash

Pre-capture user-space registers at `sys_enter_*` tracepoints (where `pt_regs` is guaranteed to be the user-space snapshot), store them in a PID-indexed BPF LRU map, and retrieve them at `sched_switch`. Use a hash of 3 RBP-chained frame pointers as a "stack signature" to throttle full stack uploads.

**Pros:** Avoids reading user-space registers in the forbidden scheduler context; adds a deduplication mechanism to reduce bandwidth.

**Cons:** The RBP hash is completely invalid on binaries compiled with `-fomit-frame-pointer` — RBP holds arbitrary general-purpose values, producing garbage hashes that change every event, defeating the throttling mechanism entirely. The `sched_wakeup` backfill strategy for `-EFAULT` was found to have a fundamental timing error: at wakeup time, the target thread's `mm_struct` has not been switched in and swapped pages remain in swap. The approach fails on both the deduplication and the recovery fronts.

### Option D: Three-tuple (RSP, RIP, *(RSP)) + next-sched-switch backfill

Replace the RBP hash with `(RSP, RIP, *(RSP))` — the stack pointer, instruction pointer, and top-of-stack return address — which are valid regardless of frame pointer presence. Fix the backfill timing by performing recovery when the target thread is next scheduled in (as `sched_switch` next), where `mm_struct` is active and pages are in RAM.

**Pros:** The three-tuple is immune to `-fomit-frame-pointer`; the corrected backfill timing is sound.

**Cons:** The overall approach remains over-engineered with no external production validation. The `sys_enter` pre-store LRU map introduces complexity (stale registers for preempted threads, LRU eviction under high concurrency, unverified 4.18 behavior). No existing production tool uses this exact combination.

### Option E: bpf_get_stackid fast path + Elastic Stack Delta deep unwinding

Use `bpf_get_stackid(ctx, &stack_traces, BPF_F_USER_STACK)` as the default fast path (~100ns overhead, 4 bytes transmitted per event). When the stack is suspiciously shallow (<=2 frames, indicating missing frame pointers), trigger a deep unwinding path based on the Elastic/OpenTelemetry eBPF Profiler's "stack delta" technique: precompute DWARF CFI rules into compact tables loaded into BPF maps, walk frames via tail-call chains (4 frames per BPF program, up to 29 tail calls = 116 frames), reading user-space registers directly from `task_struct->stack` entry `pt_regs`. On `bpf_probe_read_user` failure, immediately stop, preserve already-collected frames, and mark the event with `FLAG_PARTIAL_STACK`.

**Pros:** Production-validated at scale — Elastic runs this on millions of nodes. The fast path has near-zero overhead for frame-pointer-enabled binaries. Deep unwinding transmits frame address arrays (~256 bytes), not raw stack memory. Register retrieval from `task_struct` entry `pt_regs` works for all off-CPU scenarios including preemption. Graceful degradation on page faults is proven sufficient in production. Works on 4.15+ kernels. No additional probes or LRU maps needed.

**Cons:** Stack delta table precomputation adds startup latency. The Elastic implementation is in Go/C, requiring Rust porting effort. On 4.18 kernels, tail-call programs must stay within 4,096 instructions each, requiring careful splitting.

## Analysis

The design underwent five distinct versions across the spec chain, with each of versions A through D being found to have fundamental flaws during expert review.

Version A was rejected in the first design review for violating two hard kernel constraints: `bpf_loop` unavailability on 4.18 and the 4,096-instruction limit making a 20-layer unrolled DWARF walk infeasible.

Version B was proposed as the corrective response, moving all DWARF logic to userspace. The second expert review identified that at production event rates, the raw stack bandwidth (~50 MB/s) would overwhelm the ringbuf, and that user-space page faults during `sched_switch` could not be reliably worked around.

Version C introduced a sys_enter pre-store mechanism and RBP-based stack signature for throttling. The third expert review panel (three independent reviewers: eBPF, algorithm, and architecture) identified two fatal errors: (1) RBP is a general-purpose register under `-fomit-frame-pointer`, making the hash meaningless on x86_64; (2) the `sched_wakeup` backfill assumes the target thread's address space is active, which is false — wakeup only places the thread on the run queue without switching `mm_struct`.

Version D corrected both errors (three-tuple signature, next-sched-switch backfill) but remained an over-engineered combination with no external validation.

Version E emerged from independent research into the Elastic/OpenTelemetry eBPF Profiler's production approach. It provides a two-layer design (fast path + deep unwinding) with millions-of-nodes production validation, eliminates the need for extra probes or LRU maps, and handles page faults through proven graceful degradation rather than complex backfill mechanisms.

## Decision

Adopt Option E: `bpf_get_stackid` fast path combined with the Elastic Stack Delta deep unwinding approach. This is a two-layer strategy where the fast path handles the common case (frame-pointer-enabled binaries) with near-zero overhead, and the deep path handles `-fomit-frame-pointer` binaries using precomputed DWARF tables and BPF tail-call chains.

The decision is driven by three factors: (1) production validation at massive scale eliminates the risk of theoretical feasibility gaps that doomed versions A-D; (2) the two-layer design provides optimal overhead characteristics — the vast majority of events use the cheap fast path; (3) graceful degradation on page faults avoids the complexity of backfill mechanisms that proved error-prone in versions C and D.

## Consequences

- **Architecture:** The userspace control plane must implement `.eh_frame` parsing and stack delta table generation at startup. A `hash-of-maps` BPF map structure organizes delta tables by executable. Tail-call maps must be set up for the deep unwinding program chain.
- **Overhead:** Fast path adds ~100ns and 4 bytes per event. Deep path adds ~2-5 microseconds and ~256 bytes per event, triggered only when frame pointers are absent.
- **Compatibility:** Works on 4.15+ kernels (tail calls available since 4.2). On 4.18, each tail-call program must stay within 4,096 instructions, requiring careful splitting of the unwinding logic.
- **Risk:** Porting the Elastic stack delta implementation from Go/C to Rust is non-trivial engineering. The startup cost of scanning `/proc/PID/maps` and parsing `.eh_frame` sections adds latency before data collection begins.
- **Simplification:** Eliminates the `sys_enter` pre-store LRU map, the RBP/three-tuple signature mechanism, and the backfill state machine — removing three sources of complexity and potential bugs.
