# ADR-001: Offline CLI Model vs Backend Service

- **Status:** Accepted
- **Date:** 2026-03-17

## Problem

The wPerf project needed a definitive runtime architecture to frame all subsequent implementation decisions. Two fundamentally incompatible models had been proposed across the early design documents: a long-running web backend exposing REST/GraphQL APIs (hexagonal architecture with Axum), and an offline CLI tool that records events and generates static reports. The two models were never reconciled, causing confusion about module interfaces, data flow direction, deployment model, and the entire user experience.

## Options Considered

### Option A: Hexagonal Architecture + Axum Web Backend

An always-on backend service using Rust's Axum framework to expose REST and GraphQL APIs. The frontend (Excalidraw-based) would query the backend for data, and symbol resolution would happen on-demand when users click edges in the UI.

**Pros:**
- Supports interactive, real-time exploration of profiling data
- On-demand symbolization defers expensive addr2line work until actually needed
- Familiar microservice deployment model for cloud-native teams

**Cons:**
- Requires a running server process on or near the target machine, increasing deployment complexity
- REST/GraphQL API surface increases attack surface and maintenance burden
- Contradicts the use case of profiling production servers where installing a web service is undesirable
- Violates the offline-analysis requirement: production servers and air-gapped networks cannot serve web UIs
- CDN dependencies for frontend assets (ECharts, Virgil font, Google Fonts) break offline scenarios entirely

### Option B: Offline CLI (record/report model, like perf and KUtrace)

A single binary with subcommands (`wperf record`, `wperf report`) that captures events to a binary `.wperf` file, then produces a fully self-contained HTML report offline. No backend service, no network dependencies.

**Pros:**
- Aligns perfectly with the `perf record` / `perf report` mental model that Linux performance engineers already know
- Consistent with the KUtrace offline processing philosophy referenced in the project requirements
- Single binary with clap subcommands guarantees version consistency between record and report
- Self-contained HTML (all JS, CSS, fonts Base64-inlined) works in any air-gapped environment
- No deployment overhead: copy binary, run, transfer `.wperf` file, analyze elsewhere

**Cons:**
- No real-time interactive exploration; analysis is post-hoc
- Symbol resolution must happen at report time, not on-demand
- Large HTML files if many edges/flamegraphs are embedded

## Analysis

The design review (C-5 finding) identified a root contradiction between the two architectural models. The earlier design document described a hexagonal architecture with an Axum backend exposing REST/GraphQL APIs and on-demand symbolization. The subsequent implementation plan described a pure offline tool producing static HTML. The review classified this as Critical, noting: "This is not design evolution but an unreconciled fundamental divergence. It will cause the development team to have inconsistent understanding of the target architecture."

The review explicitly recommended the static HTML route, citing alignment with KUtrace's design philosophy and deployment simplicity. The design team's reply fully accepted this finding, acknowledging that the Axum backend was a legacy of earlier microservice-influenced thinking, while the offline model reflected the later, definitive direction.

The secondary review reinforced this decision but recommended consolidating from two separate binaries (`wperf-record` and `wperf-report`) into a single binary with subcommands, avoiding version mismatch risk and reducing maintenance cost. This aligns exactly with how `perf` works (`perf record`, `perf report`, `perf stat` are all subcommands of one binary).

The pre-execution readiness audit confirmed that the original archived project suffered from a 5-layer architecture (Frontend, Backend, Analyzer, Collector, BPF) with 4 transformation boundaries, contributing directly to the "S-curve collapse" where 64% of the last 66 commits were fixes. Simplifying to an offline CLI eliminates two entire layers (Frontend runtime, Backend service).

## Decision

**Option B: Offline CLI** was chosen, implemented as a single `wperf` binary with `record` and `report` subcommands.

The hexagonal architecture and Axum backend are permanentlyDeprecated. The final architecture is:
- `wperf record [options]` — BPF probe attachment, event capture to `.wperf` binary file
- `wperf report [options]` — offline analysis pipeline, self-contained `.html` output
- `wperf version` — version information

All frontend resources (ECharts JS, Virgil woff2 font, JetBrains Mono woff2 font, CSS) are Base64-inlined at Rust build time via `include_str!`, producing a single `.html` file that works offline in any browser.

## Consequences

**Architecture:**
- Eliminates the Axum web framework dependency and all REST/GraphQL API surface
- Removes the need for a long-running server process on production machines
- Introduces a new `.wperf` binary file format (wPRF specification) as the data contract between record and report phases
- Single binary via clap subcommands guarantees record/report version consistency

**Implementation:**
- Symbol resolution must be performed at `wperf report` time, not on-demand; this increases report generation time but eliminates runtime overhead
- HTML generation uses Rust template engine with `include_str!` for all embedded resources
- Flamegraph SVGs are generated server-side using the `inferno` crate, requiring zero client-side JS frameworks
- Graph layout uses Dagre (Sugiyama algorithm) computed during report generation, with absolute XY coordinates fed to ECharts in `layout: 'none'` mode

**Risk:**
- The `.wperf` binary format becomes a critical interface; forward compatibility must be carefully designed (TLV encoding with unknown-section skip, crash recovery via `data_section_end_offset` in the 64-byte header)
- Large profiling sessions may produce large HTML files; SVG body size is capped at 5MB with automatic downsampling
