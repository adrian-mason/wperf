# ADR-016: Retire I-1 Conservation Invariant, Narrow I-6 Depth Monotonicity

- **Status:** Proposed
- **Date:** 2026-03-31
- **Supersedes:** ADR-015's per-entry-edge conservation definition (§35-60); ADR-007/ADR-012/final-design invariant count (7→6) and I-6 scope.
- **Amends:** ADR-007, ADR-012, final-design.md

## Problem

ADR-015 redefined I-1 from global sum equality to per-entry-edge conservation:

> ∀ entry edge e: e.attributed_delay + Σ(child.attributed_delay for child in cascade_subtree(e)) == e.raw_wait

During post-merge review of PR #65/#66, @Challenger proved this definition does not hold for fan-out graphs using the BUG-3 regression test case.

### Evidence: BUG-3 counterexample

Graph: `A→B [0,100), B→C [0,60), B→D [20,80)`

| Edge | raw_wait | attributed_delay |
|------|----------|-----------------|
| A→B  | 100      | 20              |
| B→C  | 60       | 60              |
| B→D  | 60       | 60              |

Per ADR-015 definition, entry edge A→B's subtree sum = 20 + 60 + 60 = 140 ≠ 100.

### Root cause

The cascade algorithm computes each edge **independently** (engine.rs line 39-51: outer loop over `graph.all_edges()`). B→C's `attributed_delay=60` comes from B→C's own independent cascade computation (raw=60, leaf), not from A→B's cascade. ADR-015's definition sums `attributed_delay` values from **different independent cascade computations** as if they belonged to a single cascade tree.

Within A→B's own cascade computation, `compute_cascade(B, [0,100))` returns `(propagated=80, self_blame=20)`, and `propagated + self_blame = 100 = raw_wait` — but this is internal to the recursive call and equals `attributed + propagated = raw_wait` by construction (`attributed = raw_wait - propagated`), making it a tautology that cannot catch bugs.

### Conclusion (Oracle ruling, 2026-03-31)

Per-edge independent cascade has **no cross-edge conservation invariant**. Each edge is cascaded independently; there is no shared "conservation pool" across edges. A distinct I-1 (stronger than I-2, weaker than impossible global sum) does not exist for this algorithm architecture.

This is the same structural issue as ADR-015's predecessor: the original global sum equality was impossible because per-edge independent cascade absorbs weight at intermediate nodes. Per-entry-edge subtree-sum equality is impossible for the same reason — downstream edges have independent cascade computations with different raw_wait values.

### I-6 depth monotonicity

Separately, PR #65's cascade recursion fix (returning `(propagated, self_blame)` tuple) revealed that I-6 (depth monotonicity: `total_attributed(deep) ≤ total_attributed(shallow)`) does not hold in general. When `target_count > 1` or `external_waiters > 1`, integer division causes `child_absorbed < window.duration()`, so deeper recursion may propagate **less** weight than the depth-truncation base case `(0, window.duration())`.

I-6 holds for simple chains (no fan-out, no concurrent waiters) and is verified by unit tests on such graphs. It does not hold for general graphs and cannot be included in property-based testing over random graphs.

## Decision

### 1. Retire I-1 as a separate invariant

I-1 (conservation) is removed from the invariant set. The invariant count changes from 7 to 6: {I-2, I-3, I-4, I-5, I-6*, I-7}.

The `is_conserved` name is **deprecated**. It has undergone three semantic shifts (global sum equality → per-entry-edge subtree sum → non-amplification alias) and is no longer recoverable as a clear API contract.

### 2. Narrow I-6 to qualified invariant

I-6 (depth monotonicity) is retained as a **qualified invariant**: it holds for simple chains only (no fan-out, no concurrent waiters). It is verified by unit tests on simple chain graphs. It is excluded from property-based testing on random graphs.

### 3. Redefine production sentinel

The production release sentinel is renamed from `is_conserved` to `invariants_ok` and defined as:

```
invariants_ok = I-2 (non-amplification) ∧ I-7 (locality)
```

This composite flag:
- Runs in ALL builds (release + debug)
- Is the only runtime invariant check in release mode
- Is exported in JSON output as `invariants_ok` (replacing `is_conserved`)
- Serves as a **structural postcondition guard** (no overflow, topology preserved), not a bug detector

**Limitation:** I-2 (non-amplification) is unfalsifiable by construction through the current cascade path. `engine.rs:51` computes `attributed = raw_wait.saturating_sub(propagated)`, which guarantees `attributed ≤ raw_wait` for any value of `propagated`. I-7 (locality) checks topology/input preservation only, not attribution correctness. Therefore `invariants_ok` catches **0/5 known bugs** — all 5 bugs produce wrong `attributed` values that still satisfy both I-2 and I-7.

Known-bug detection relies on the full verification stack: targeted regression tests, property-based testing (I-3/I-4/I-5), differential oracle (Rust vs Python), and mutation testing. The sentinel's value is defense-in-depth against future regressions that might bypass `saturating_sub` or corrupt graph topology.

### 4. Updated verification coverage matrix

| Verification layer | Invariants checked | Scope |
|---|---|---|
| **Release sentinel** (`invariants_ok`) | I-2 ∧ I-7 | All builds, all graphs |
| **Debug assertions** | I-3, I-4 | Debug builds only |
| **Property-based (10K)** | I-2, I-3, I-4, I-5, I-7 | Random graphs |
| **Unit tests** | I-6 | Simple chains only |
| **Differential testing** | Rust vs Python agreement | 6 fixed scenarios |

### 5. Phase 0 gate update

Phase 0 exit criteria change from:
- `assert_entry_edge_conserved()` 0 violations

To:
- `invariants_ok` (I-2 ∧ I-7) 0 violations on all test graphs
- I-2 through I-5, I-7 verified by 10K proptest with 0 violations
- I-6 verified by unit tests on simple chains
- 5 bug regressions pass
- Differential testing (Rust vs Python) ≤1.0ms
- Mutation testing ≥90% kill rate

## What changes

### Spec updates required

- **ADR-015:** Status changed to "Superseded by ADR-016". Per-entry-edge subtree-sum definition retracted.
- **ADR-007:** Invariant table updated (7→6, I-6 qualified). Layer 1 sentinel renamed. Layer 3 coverage updated.
- **ADR-012:** Conservation characterization updated. Invariant count updated. `is_conserved` references updated to `invariants_ok`.
- **final-design.md §3.4:** Invariant table updated. I-1 removed. I-6 qualified. Sentinel renamed.
- **final-design.md §7.3:** Phase 0 gate criteria updated per §5 above. Phase 1/2a/2b gate table rows: `is_conserved` → `invariants_ok`.

### Code updates required (separate PR)

- `is_conserved()` → `invariants_ok()`: checks I-2 ∧ I-7 (restore I-7 check removed in PR #66)
- `verify_conservation()` → `verify_engine_postconditions()`
- `GraphMetrics.is_conserved` → `GraphMetrics.invariants_ok`
- JSON output field renamed: `"is_conserved"` → `"invariants_ok"`
- `ConservationError` → `InvariantError` (or similar)

### Migration strategy

The `is_conserved` name is deprecated with **no alias or transition period**. This is a one-shot breaking rename:

- **Justification:** The project is pre-1.0 with no external consumers. There is no compatibility guarantee to honor.
- **Scope:** All renames (function, struct field, JSON field, error type) land in a single code alignment PR.
- **No alias:** No `is_conserved` backward-compatible field/function is retained. The name has undergone three semantic shifts (global sum → subtree sum → non-amplification alias) and is not recoverable.
- **Downstream impact:** `docs/` examples, `tests/` assertions, and `output.rs` JSON serialization are updated in the same PR. No separate consumer migration needed.

## Consequences

**Risk:** The invariant set is weaker than originally designed. The removed invariant (I-1) was impossible to satisfy — it was vacuously strong. The remaining invariants are the strongest correct properties for per-edge independent cascade.

**Limitation — sentinel is structural, not diagnostic:** `invariants_ok` (I-2 ∧ I-7) catches 0/5 known bugs. I-2 is unfalsifiable by construction (`saturating_sub` guarantees `attributed ≤ raw`). I-7 checks topology preservation only. The sentinel's purpose is defense-in-depth against future code changes that might bypass `saturating_sub` or corrupt graph topology — it is not a bug detector. Known-bug coverage comes entirely from the full verification stack (regression tests, proptest, differential oracle, mutation testing).

**Phase 1 implication:** The Phase 1 gate criterion `invariants_ok==true on real BPF data` is necessary but extremely weak for validating BPF→cascade correctness. Actual correctness signal for real BPF data will come from differential testing, regression tests with known-good BPF event sequences, and `drop_count` metrics for data completeness. Phase 1 planning must not over-rely on the sentinel flag.

**Mitigation:** The 5-layer verification pyramid remains intact. Differential testing (Rust vs Python oracle) provides cross-implementation consistency checking independent of invariant definitions, though it cannot rule out common-mode misunderstandings shared by both implementations. The 5 bug regression tests directly verify correctness on known failure modes.

**Benefit:** The spec now accurately describes what the algorithm can guarantee. No more semantic drift between spec promises and code reality.
