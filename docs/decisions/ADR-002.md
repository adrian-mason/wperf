# ADR-002: Kernel Compatibility — Dynamic Feature Probing vs Static Tiers

- **Status:** Accepted
- **Date:** 2026-03-17

## Problem

wPerf-rs targets a wide range of Linux kernel versions, from RHEL 8 (kernel 4.18) to modern mainline 6.x kernels. Key eBPF features required by the tool — ring buffer, bpf_loop, BTF/CO-RE, cgroupv2 — were introduced at different kernel versions. The project needed a strategy to handle this heterogeneous feature landscape without maintaining separate codebases or silently failing on older kernels.

The initial design proposed static "tiers" (Tier 1: 5.17+, Tier 2: 5.8-5.16, Tier 3: 4.18-5.7) that bundled features by kernel version range. However, real-world kernel deployments invalidate this model.

## Options Considered

### Option A: Static Tier 1/2/3 by Kernel Version

Define three static tiers based on `uname -r` version parsing. Each tier maps to a fixed feature set. Compile separate `.bpf.o` files per tier, and the user-space loader selects the appropriate one at startup.

**Pros:**
- Conceptually simple: one version check determines the entire feature set
- Clear documentation: users know exactly what their kernel version supports
- Predictable behavior per tier

**Cons:**
- Enterprise distributions backport features aggressively: RHEL 8 (nominally 4.18) can have BTF support starting from 8.2+, while a custom-compiled 5.x kernel may lack `CONFIG_DEBUG_INFO_BTF`
- Ring buffer (5.8+) and bpf_loop (5.17+) are orthogonal features; a kernel at 5.10 with a backported bpf_loop would be misclassified
- Requires maintaining 3 separate BPF compilation targets with `#define` isolation, doubling or tripling build complexity
- Version parsing is fragile: RHEL kernel strings like `4.18.0-477.10.1.el8_8.x86_64` encode backported features that version comparison cannot detect
- Any new feature requires updating the tier definitions

### Option B: Dynamic Per-Feature Probing at Startup

At program startup, between BPF skeleton `open()` and `load()`, individually probe each feature using the standard libbpf "probe, reconfigure, load" pattern. Each feature is tested independently; the program adapts its configuration based on what actually works on the running kernel.

**Pros:**
- Correctly handles backported features (RHEL 8 with BTF, Ubuntu HWE kernels with newer BPF features)
- Correctly handles stripped features (custom kernels compiled without specific configs)
- Single `.bpf.o` file using CO-RE conditional paths (`bpf_core_type_exists`, `bpf_core_field_exists`)
- Matches the proven pattern used by bcc/libbpf-tools production tools (filelife, opensnoop, biosnoop, mountsnoop, tcppktlat)
- Feature matrix is extensible: adding a new feature probe requires only adding a new check, not restructuring tiers

**Cons:**
- Startup time increases slightly due to probe attempts (each `bpf_map_create` probe takes ~1ms)
- More complex startup logic with multiple conditional paths
- Harder to document: the supported feature set depends on the running kernel, not a simple version table
- Risk of untested feature combinations

## Analysis

The initial design review (C-1 finding) exposed the fundamental flaw in the original design: it claimed ring buffer support on kernel 4.18, which is impossible since `BPF_MAP_TYPE_RINGBUF` was introduced in Linux 5.8. This was classified as Critical because "loading the BPF program on 4.18 would directly fail with `EINVAL`." The review proposed either a transport degradation strategy or raising the minimum kernel version to 5.8.

The design team's reply accepted the finding and proposed an `EventTransport` trait with two implementations. However, the secondary review identified a deeper problem: the "dual `.bpf.o`" compilation approach (one for ringbuf, one for perfarray) would produce "two skeleton files, two build artifacts, `#define` isolation prone to missed changes, and two Rust-side loading paths."

Research into production BPF tooling revealed the industry consensus. Analysis of bcc/libbpf-tools (with specific citations to filelife, opensnoop, biosnoop, mountsnoop, tcppktlat — five production-validated tools) confirmed that the standard pattern is per-feature independent probing, not version-based tiering. The `compat.bpf.h` pattern from bcc/libbpf-tools demonstrates a single-ELF approach where both ringbuf and perfarray maps are declared in the same object file, with runtime CO-RE conditionals selecting the active path.

The key technical insight came from examining real-world kernel deployments: RHEL 8 (nominally 4.18) can backport BTF starting from 8.2+, while a custom-compiled 5.x kernel may lack `CONFIG_DEBUG_INFO_BTF` entirely. Ring buffer availability (5.8+) and bpf_loop availability (5.17+) are completely orthogonal features. Static tiers that bundle these features together by version range produce incorrect classifications for a significant fraction of real deployments.

For structure field versioning (e.g., `task_struct.state` renamed to `task_struct.__state`, `bio.bi_disk` to `bio.bi_bdev`), the bcc/libbpf-tools `core_fixes.bpf.h` pattern provides runtime field selection via `bpf_core_field_exists()` with zero runtime overhead (resolved at BPF load time by the CO-RE relocator).

## Decision

**Option B: Dynamic per-feature probing** was chosen, following the libbpf ecosystem's standard "probe, reconfigure, load" pattern.

A single `wperf.bpf.o` ELF file is compiled with all features declared. At startup, before `skeleton.load()`, each feature is individually probed:

| Feature | Probe Method | Degradation Action |
|---------|-------------|-------------------|
| ringbuf | `probe_ringbuf()`: attempt `bpf_map_create(BPF_MAP_TYPE_RINGBUF)` | Reconfigure events map to `PERF_EVENT_ARRAY`; disable heap map via `set_autocreate(false)` |
| BTF (vmlinux) | Check `/sys/kernel/btf/vmlinux` existence | Try embedded BTF fallback; if still fails, use hardcoded offsets |
| bpf_loop | `libbpf_probe_bpf_helper(TRACEPOINT, BPF_FUNC_loop)` | Fall back to `#pragma unroll` bounded loop |
| cgroupv2 | Check `/sys/fs/cgroup/cgroup.controllers` | Disable cgroup filtering; `--cgroup` flag errors out |
| tracepoint existence | Check `/sys/kernel/tracing/events/{cat}/{name}` | `bpf_program__set_autoload(prog, false)` |
| kprobe existence | Check blacklist + `available_filter_functions` | `set_autoload(false)` |
| fentry | Actual attach test | Fall back to kprobe/tracepoint |

The static tier table is retained only as documentation ("typical kernel capabilities reference"), never as runtime logic.

## Consequences

**Architecture:**
- Single BPF object file simplifies the build system and eliminates divergent code paths
- BPF-side code uses `bpf_core_type_exists(struct bpf_ringbuf)` for conditional transport selection and `bpf_core_field_exists()` for struct field versioning — both resolved at load time with zero runtime overhead
- User-space `EventTransport` trait abstracts the consumption-side differences between ringbuf and perfarray

**Implementation:**
- The perfarray degradation path requires a Min-Heap Reorder Buffer (50ms tolerance window) in user space to restore global event ordering, since perfarray is per-CPU with no cross-CPU ordering guarantee
- Each probe adds approximately 1ms to startup time; total probe overhead is under 10ms for the full matrix
- The `heap` percpu-array map serves as a staging area for `bpf_perf_event_output` in the perfarray path (since perf output requires a pointer to pre-filled data, unlike ringbuf's reserve/submit model)

**Risk:**
- Untested feature combinations are possible (e.g., ringbuf available but BTF unavailable). The feature probing code must be integration-tested against representative kernel configurations
- Timestamp monotonicity across CPUs is not guaranteed by `bpf_ktime_get_ns()` during NTP clock slope updates. For the perfarray path, the 50us false-wakeup pruning threshold and best-effort state machine tolerance mitigate this. The ringbuf path is unaffected because its spinlock-serialized reservation provides global consumption order
