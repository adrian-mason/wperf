# ADR-014: libbpf/CO-RE Best Practices Alignment (Nakryiko Reference)

- **Status:** Accepted
- **Date:** 2026-03-18

## Problem

wPerf's eBPF probe layer is built on libbpf with CO-RE (Compile Once — Run Everywhere). Andrii Nakryiko, the libbpf maintainer, has published authoritative guidance on BPF best practices through his blog (nakryiko.com), covering CO-RE patterns, ring buffer usage, skeleton lifecycle, portability, and common anti-patterns. The wPerf design should be validated against this authoritative source to ensure alignment and identify any gaps.

This ADR documents the validation results and records specific recommendations that affect wPerf's implementation.

## Analysis Method

All articles on nakryiko.com were reviewed, with focus on:
- BPF CO-RE reference guide
- BPF ring buffer design and usage
- BPF portability and CO-RE
- Building BPF applications with libbpf-bootstrap
- Journey to libbpf 1.0
- BTF deduplication and Linux kernel BTF
- BPF tips and tricks

Each recommendation was compared against the current wPerf design (final-design.md, ADR-002, ADR-004) and classified as aligned, gap, or enhancement opportunity.

## Alignment Confirmation

The following design decisions are confirmed as correctly aligned with Nakryiko's authoritative guidance:

### CO-RE Patterns (All Aligned)

| wPerf Design Element | Nakryiko Recommendation | Status |
|---|---|---|
| Dynamic per-feature probing over version tiers (ADR-002) | "Use `bpf_core_type_exists()`, `bpf_core_field_exists()` instead of kernel version comparisons. Kernel version checks fail when features are backported" | Aligned |
| `___old`/`___new` suffix convention for struct versioning (§ 1.3) | "Define alternative type versions with triple-underscore suffixes. libbpf matches whichever definition kernel BTF contains" | Aligned |
| `bpf_core_field_exists()` for field rename handling (§ 1.3) | "Guard optional fields with existence checks. The verifier eliminates dead code paths at load time with zero runtime overhead" | Aligned |
| Single-ELF with CO-RE conditionals (ADR-004) | Single BPF object with `bpf_core_type_exists(struct bpf_ringbuf)` for transport selection | Aligned |
| `const volatile bool` feature flags for auxiliary probes (§ 2.1) | "`const volatile` is required to prevent Clang from optimizing away the variable. Must be set before `skeleton.load()`" | Aligned |
| Partial struct definitions with `preserve_access_index` (§ 1.3) | "Partial struct definitions are sufficient for CO-RE relocation. Field order is irrelevant" | Aligned |

### Ring Buffer Patterns (All Aligned)

| wPerf Design Element | Nakryiko Recommendation | Status |
|---|---|---|
| `reserve_buf()`/`submit_buf()` over `bpf_ringbuf_output()` (ADR-004) | "The reserve/submit API is superior. If reserve returns non-NULL, submit will always succeed. Avoids intermediate copy" | Aligned |
| `set_autocreate(false)` for unused heap map (ADR-004) | "Use `bpf_map__set_autocreate(map, false)` for optional maps" | Aligned |
| BPF-side `drop_counter` for ringbuf loss detection (ADR-012) | "Unlike perfbuf, ringbuf does not provide an automatic callback for lost samples. Drop detection must be handled explicitly from BPF side" | Aligned |
| CPU ID embedded in BaseEvent struct (§ 2.4) | "Ring buffer consumer callback does not receive CPU index. CPU identification must be embedded in the event data" | Aligned |
| Perfarray staging via percpu-array heap (ADR-004) | "`bpf_perf_event_output` requires a pointer to pre-filled data, unlike ringbuf's reserve/submit model" | Aligned |

### Skeleton and Build Patterns (All Aligned)

| wPerf Design Element | Nakryiko Recommendation | Status |
|---|---|---|
| open → probe → reconfigure → load → attach lifecycle (ADR-002) | "Follow the open/load/attach lifecycle strictly. Read-only globals must be set between open and load" | Aligned |
| `set_autoload(false)` for missing probes (ADR-002) | "Use `bpf_program__set_autoload(prog, false)` for optional programs" | Aligned |
| Tracepoints for syscall interception (§ 2.1 `tp/syscalls/sys_enter_futex`) | "Prefer tracepoints over kprobes for syscalls" (syscall naming changed in 4.17) | Aligned |

## Gaps Requiring Correction

### Gap 1: BTF Fallback (Correction from ADR-002-supplement)

**Nakryiko:** "libbpf transparently handles BTF compatibility with older kernels, stripping unsupported features without user intervention."

**Current design (ADR-002):** States "hardcoded offsets" as the final BTF fallback.

**Correction:** As documented in ADR-002-supplement, the actual fallback is embedded BTF or failure — not hardcoded offsets. libbpf's auto-sanitization of BTF for older kernels provides additional resilience but does not eliminate the BTF requirement for CO-RE programs.

### Gap 2: Standard Tracepoints for Scheduler Events (Correction from ADR-013)

**Nakryiko:** CO-RE reference guide documents `BPF_CORE_READ()` on `task_struct` as the standard pattern for accessing task metadata in BPF programs.

**Current design (§ 2.1):** Uses `tp/sched/sched_switch` which provides only pre-formatted scalar fields, not `task_struct` pointers.

**Correction:** As documented in ADR-013, scheduler probes should use `tp_btf` with `raw_tp` fallback to get `task_struct *` access for `tgid` and other metadata.

## Enhancement Opportunities

These are not conflicts with the current design but actionable improvements identified from Nakryiko's guidance:

### E1: Ringbuf Notification Flags for Throughput Tuning

**Source:** "BPF ring buffer" article.

**Recommendation:** Use `BPF_RB_NO_WAKEUP` and `BPF_RB_FORCE_WAKEUP` flags to control in-kernel notification signaling. "For high-throughput scenarios, the biggest overhead comes from in-kernel signaling of data availability."

**Impact on wPerf:** For sched_switch events at high frequency (100K+/sec), notification batching could help meet the <3% CPU overhead gate. Strategy: use `BPF_RB_NO_WAKEUP` for most events, `BPF_RB_FORCE_WAKEUP` every Nth event.

**Recommendation:** Defer to Phase 2 performance tuning. Document as a known optimization lever.

### E2: Kconfig Externs for BPF-Side Version Detection

**Source:** CO-RE reference guide.

**Recommendation:** `extern int CONFIG_HZ __kconfig;` and `extern u32 LINUX_KERNEL_VERSION __kconfig;` allow BPF programs to access kernel compile-time configuration. The verifier treats these as constants for dead code elimination.

**Impact on wPerf:** Complements user-space probing with BPF-side detection. Example: `CONFIG_HZ` could inform timer-related calculations in the BPF code. `LINUX_KERNEL_VERSION` could gate features that are version-specific within the BPF program itself.

**Recommendation:** Add to the BPF source template. Low cost, high utility.

### E3: User-Space Ringbuf Sizing via `bpf_map__set_max_entries()`

**Source:** "BPF ring buffer" article.

**Recommendation:** Ring buffer size can be specified in BPF code and overridden from user-space before load.

**Impact on wPerf:** The `wperf record` CLI should accept a `--buffer-size` flag that calls `bpf_map__set_max_entries()` on the events map between open and load. Size must be a power of 2 and a multiple of page size.

**Recommendation:** Add to Phase 1 CLI specification.

### E4: Verifier Log Capture for Debugging

**Source:** "Journey to libbpf 1.0".

**Recommendation:** Use `kernel_log_buf`, `kernel_log_size`, `kernel_log_level` open options to capture BPF verifier logs.

**Impact on wPerf:** A `wperf record --debug-bpf` flag that enables verifier log capture would help diagnose load failures on unfamiliar kernels. Critical for field support.

**Recommendation:** Add to Phase 1 as a debug flag.

### E5: `bpf_core_enum_value_exists()` for Helper Detection

**Source:** CO-RE reference guide.

**Recommendation:** `bpf_core_enum_value_exists(enum bpf_func_id, BPF_FUNC_ringbuf_reserve)` detects optional BPF helpers at load time. More reliable than kernel version checking.

**Impact on wPerf:** Can be used as a BPF-side complement to user-space `probe_ringbuf()`. The current design's `bpf_core_type_exists(struct bpf_ringbuf)` pattern is already validated, but `bpf_core_enum_value_exists()` provides an alternative detection mechanism.

**Recommendation:** Consider for `bpf_loop` detection (ADR-002's `bpf_loop` fallback to `#pragma unroll`). BPF-side detection via `bpf_core_enum_value_exists(enum bpf_func_id, BPF_FUNC_loop)` could complement the user-space `libbpf_probe_bpf_helper()` check.

## Anti-Patterns to Enforce

The following anti-patterns from Nakryiko's articles must be avoided in wPerf implementation:

| Anti-Pattern | Rule | Enforcement |
|---|---|---|
| Kernel version checks for feature detection | Never use `uname -r` parsing for runtime feature decisions | Code review; ADR-002 |
| `SEC("maps")` legacy format | Always use `SEC(".maps")` with BTF-defined maps | Compiler check; libbpf 1.0 rejects legacy |
| `bpf_ringbuf_output()` for data transfer | Always use `bpf_ringbuf_reserve()`/`bpf_ringbuf_submit()` | Code review |
| Individual kernel headers in BPF source | Use `vmlinux.h` + `bpf_helpers.h` + `bpf_core_read.h` + `bpf_tracing.h` | Build system |
| `bpf_printk()` in production code | Gate behind `const volatile bool debug_verbose = false` | Code review |
| Struct-level copies of kernel types | Read only needed primitive fields via `BPF_CORE_READ()` | Code review |
| Duplicate global variable declarations | Access BPF globals only through skeleton `rodata`/`bss`/`data` sections | Code review |
| `static` keyword on mutable BPF globals | Use non-static for skeleton-accessible globals; `const volatile` for read-only config | Code review |

## Decision

The wPerf design is **strongly aligned** with Nakryiko's authoritative guidance on all critical architectural decisions (CO-RE patterns, ring buffer dual-mode, skeleton lifecycle, feature probing). Two gaps have been identified and corrected in ADR-002-supplement and ADR-013 respectively.

Five enhancement opportunities (E1-E5) are recorded for implementation planning:
- **Phase 1:** E3 (buffer sizing CLI flag), E4 (debug-bpf flag), E2 (Kconfig externs)
- **Phase 2:** E1 (notification tuning), E5 (enum value detection)

The anti-pattern table serves as a code review checklist for all BPF source contributions.

## Consequences

1. **Build system:** Must generate `vmlinux.h` from kernel BTF (via `bpftool btf dump`) or use a pre-generated version from the libbpf/vmlinux.h repository. Individual kernel headers are not used in BPF source.

2. **BPF source template:** The standard include set for all `.bpf.c` files is:
   ```c
   #include "vmlinux.h"
   #include <bpf/bpf_helpers.h>
   #include <bpf/bpf_core_read.h>
   #include <bpf/bpf_tracing.h>
   ```

3. **Shared type definitions:** Event structs shared between BPF C and Rust user-space must be defined in a common header (e.g., `wperf.h`), included by the `.bpf.c` source and processed by `bindgen` or manually mirrored in Rust. This follows the libbpf-bootstrap `<app>.h` pattern.

4. **Code review checklist:** The anti-pattern table from this ADR should be referenced during code review of all BPF-related changes.

5. **License:** BPF source must declare `char LICENSE[] SEC("license") = "Dual BSD/GPL";` — required by the kernel for `bpf_get_stackid()` and other GPL-only helpers.
