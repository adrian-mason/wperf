# ADR-011: Phase Structure — 16-Week Gated Phases vs 12-Week 6-Sprint

- **Status:** Accepted
- **Date:** 2026-03-17

## Problem

The wPerf-rs project requires a development schedule and phase structure that can deliver a production-quality off-CPU performance analysis tool while avoiding the catastrophic failure modes observed in the archived predecessor project. The archived project collapsed under integration pressure in its final days, producing an unreliable codebase despite months of development effort. The choice of phase structure directly determines whether the new project can detect and contain defects early or whether it will repeat the same terminal integration crisis.

## Options Considered

### Option A: 12-Week 6-Sprint Model (Original Plan)

The original project setup defined six two-week sprints running sequentially across 12 weeks:

- S1 (Weeks 1-2): Algorithm core -- Cascade Engine, Tarjan SCC, Condensation DAG, Critical Path DP
- S2 (Weeks 3-4): Data format (wPRF4) and CLI skeleton
- S3 (Weeks 5-6): eBPF probes and transport layer
- S4 (Weeks 7-8): Userspace collector, reorder buffer, event correlation state machine
- S5 (Weeks 9-10): Frontend visualization -- Dagre layout, ECharts rendering, SVG flamegraph
- S6 (Weeks 11-12): Integration, cross-kernel testing, v0.1.0-alpha release

**Pros:**
- Shorter total duration (12 weeks vs 16 weeks)
- Familiar sprint cadence with clear deliverables per sprint
- Milestones map directly to GitHub project management tooling

**Cons:**
- No pre-validation of high-risk technical assumptions before committing to implementation
- Phase 0 (algorithm core) allocated only 2 weeks, with no time for differential testing against the Python reference implementation or mutation testing
- No explicit gate criteria between sprints -- a sprint ending does not prove the deliverables are correct
- Integration sprint (S6) compressed into 2 weeks, the same pattern that caused the archived project's collapse
- No mechanism to prevent the three failure modes observed in the archived project: spec modification to match output, cross-phase workarounds, and concurrent multi-layer development

### Option B: 16-Week Model with Gate 0 + Gated Phases

A restructured timeline adding a prototype validation week (Gate 0), expanding the algorithm phase from 2 to 3 weeks, splitting the middle phase to isolate graph-structure changes, and enforcing explicit exit gates between every phase:

- Gate 0 (Week 1): Three throwaway prototypes validating high-risk assumptions
- Phase 0 (Weeks 2-4): Cascade correctness with 5-layer verification, including differential and mutation testing
- Phase 1 (Weeks 5-8): Minimal data pipeline with early simplified visualization
- Phase 2a (Weeks 9-10): Wait-reason annotation (futex) -- no graph structure change
- Phase 2b (Weeks 11-12): Synthetic edges and pseudo-thread attribution -- changes graph structure
- Phase 3 (Weeks 13-16): Stack collection, production-grade HTML report, cross-kernel testing

**Pros:**
- Gate 0 validates that eBPF collection works, the Python Cascade reference produces correct output, and the wPRF format roundtrips correctly -- before any production code is written
- Phase 0 has sufficient time (3 weeks) for property-based testing across 10,000+ random graphs, differential testing against the Python oracle, and mutation testing targeting at least 90% detection rate
- Phase 2 split isolates risk: Phase 2a (futex annotation) does not alter the Wait-For Graph topology, while Phase 2b (synthetic edges for disk/network pseudo-threads) does -- keeping graph-structure-altering changes separate
- Phase 1 includes minimal visualization (dot SVG) to avoid the archived project's confusion between algorithm bugs and rendering bugs
- Each phase has a defined exit gate with automated pass/fail criteria
- Three explicit prohibitions address the archived project's three deadliest failure modes

**Cons:**
- 4 weeks longer total duration (16 vs 12 weeks)
- Gate 0 prototype code is discarded -- pure validation cost
- More administrative overhead in gate management

## Analysis

### Evidence from the Archived Project's Failure

The archived wPerf project produced 462 total commits, of which 198 were fix commits (42.8%). The fix-to-feature ratio of 1.51:1 is pathologically high, indicating that more effort was spent repairing defects than building features.

The commit history reveals a dramatic S-curve collapse. From February 15 through February 23, the project maintained a near-zero fix rate (0-2%), indicating rapid feature development with deferred validation. On February 24, the fix rate jumped to 17%, rising to 21% on February 25, then collapsing to 59% on February 26, 47% on February 27, and 88% on the final day (February 28). In the last 3 days alone, 66 commits were made, 42 of which were fixes (64%). On the final day, 17 commits were made, 15 of which were fixes (88%).

The fix distribution across components reveals that the damage was not localized: 8 cascade fixes, 6 knot fixes, 5 spec relaxations, 4 DWARF fixes, 3 BPF fixes, 3 pipeline fixes, and 2 frontend fixes. This cross-cutting failure pattern is directly attributable to concurrent multi-layer development, evidenced by FACB (Frontend-Analyzer-Collector-BPF) four-layer single commits with messages like "fix: algorithms, types, colors, perf" and "fix: DWARF frame bounds, transport compat, block_rq_complete type, drop order, test coverage."

The core algorithm files suffered extreme churn: `cascade.rs` was modified 44 times (averaging one modification per 34 lines of code), and `knot.rs` was modified 33 times. The algorithm codebase totaled approximately 5,800 lines across four files (`knot.rs` 1,496 lines, `cascade.rs` 1,507 lines, `graph.rs` 1,345 lines, `analysis.rs` 1,123 lines), none of which achieved stability.

Most critically, the archived project exhibited spec erosion: 9 instances of spec relaxation or removal over 7 days (February 22-28). These included relaxing matrix assertions for CI environments, lowering minimum count thresholds, removing SCC assertions entirely, and aligning bottleneck assertions with actual (incorrect) output rather than correcting the code. This created a dependency chain where untrustworthy code led to unverifiable algorithms, which led to untrustworthy tests.

At least 8 pairs of duplicate commits with identical messages were found, proving that the AI agent repeatedly lost context and redid the same work from scratch.

### Gate 0 Prototype Validation

Gate 0 requires three prototypes, each addressing a specific high-risk assumption:

1. **eBPF Minimal Collection (2 days):** Verifies that `sched_switch` + `sched_wakeup` can be captured and correctly paired on the host kernel. If this fails, the entire tool's data acquisition model is invalid.

2. **Python Cascade Understanding (1 day):** Runs the original wPerf Python analyzer (`bottleneck.py`, 808 lines) against the paper's Figure 4 input to confirm the reference implementation produces the expected output (Network receives 80ms, Parser retains 20ms, total conserved at 100ms). If this fails, the Cascade implementation will be built on a flawed understanding of the algorithm.

3. **wPRF Roundtrip (2 days):** Validates that the 64-byte header + TLV event stream format can serialize, deserialize, and survive truncation recovery. If this fails, the file format specification is ambiguous.

All prototype code is discarded. The purpose is hypothesis validation, not code production.

### Phase 0 Expansion

The original 2-week allocation for the algorithm core was insufficient. The Cascade Engine is the only component with zero third-party production implementations worldwide. Design review alone required 6 rounds and uncovered 5 algorithmic bugs in the pseudocode (BUG-1 through BUG-4 plus NEW-BUG-1). The sweep-line partition subroutine had no pseudocode even after all review rounds. The `concurrent_waiters_count` function's semantics remained undefined.

Expanding to 3 weeks allows Week 3 to be dedicated to differential testing against the Python reference (with 1.0ms tolerance) and mutation testing (targeting 95% or higher detection rate). The 5-layer verification strategy -- invariant assertions, paper scenario tests, property-based testing across 10,000+ random graphs, differential testing, and mutation testing -- cannot be compressed into 2 weeks without sacrificing the quality guarantees that are the entire point of the restructured schedule.

### Phase 2 Split

Phase 2 was split into 2a and 2b based on a critical architectural distinction: futex wait-reason annotation (Phase 2a) attaches metadata to existing edges without altering the graph topology, while synthetic edge injection for disk and network pseudo-threads (Phase 2b) fundamentally changes the graph structure by adding new nodes and bidirectional edges. Mixing these two types of changes makes it impossible to determine whether a regression was caused by annotation logic or by graph structure modification. The archived project's experience showed that such ambiguity directly feeds the fix-on-fix spiral.

### Phase 1 Minimal Visualization

Phase 1 includes a minimal visualization output (dot-format SVG) specifically to avoid repeating the archived project's inability to distinguish algorithm errors from rendering errors. The original user requirement document explicitly noted that it was unclear whether "arrows flying everywhere" was caused by the graph algorithm or the rendering engine. By providing a simple, deterministic visualization at Phase 1, algorithm correctness can be visually inspected before the full Dagre + ECharts HTML report is built in Phase 3.

### Three Prohibitions

The three prohibitions directly counter the three deadliest failure modes from the archived project:

1. **No spec modification to match output:** The archived project relaxed or removed 9 spec assertions in 7 days rather than fixing code. Under the new model, if a test fails, the defect is in the code, never in the spec.

2. **No cross-phase workarounds:** If Phase N+1 discovers a Phase N bug, the fix must be applied at Phase N, Phase N's gate must be re-passed, and only then may Phase N+1 continue. This prevents the accumulation of cross-layer workarounds that characterized the archived project's final days.

3. **No concurrent phase development:** The archived project's FACB four-layer single commits -- modifying frontend, analyzer, collector, and BPF code in a single commit -- are directly prohibited. Strict serial phase execution ensures that each layer is stable before the next begins.

## Decision

The 16-week gated model (Option B) is chosen.

The 4-week extension is justified by the archived project's failure evidence: a project that attempted to move fast without gates lost its final 9 days to an 88% fix rate and produced an untrusted codebase. The gate structure converts "hope the integration works" into "prove each layer works before building the next." Gate 0 alone -- a one-week investment in three throwaway prototypes -- prevents committing 12+ weeks of effort to assumptions that may be false.

The decision rests on a single quantified argument: the archived project's last 3 days consumed 66 commits to produce 42 fixes, achieving nothing usable. Four additional weeks of structured validation is a better investment than even 3 days of unstructured crisis repair.

## Consequences

**Architecture:** The gated structure forces a clean layering discipline. Phase 0 produces a pure-Rust algorithm library with no BPF dependencies, testable entirely through synthetic graph inputs. Phase 1 adds BPF and file I/O but keeps visualization minimal. Phase 2a and 2b separate graph-topology-preserving changes from graph-topology-altering changes. Phase 3 adds the full visualization and production polish. Each phase boundary is also an architectural boundary.

**Implementation:** Developers cannot begin eBPF probe work until the algorithm layer has passed its gate (0 conservation violations across 10,000+ random graphs, all 5 known bug regressions passing, mutation detection rate at or above 90%, differential testing within 1.0ms tolerance of the Python reference). This may feel slow during Phase 0, but the archived project's evidence shows that "fast" multi-layer development produces 42.8% fix commits and no usable output.

**Risk:** The primary residual risk is that Gate 0 prototypes may reveal a fundamental assumption failure (e.g., eBPF event pairing does not work on the target kernel, or the Python reference implementation itself contains bugs that corrupt the differential testing oracle). However, discovering such failures in Week 1 at a cost of discarded prototype code is categorically preferable to discovering them in Week 10 after building on a flawed foundation. The differential testing oracle risk is explicitly acknowledged: if Rust and Python disagree but Rust satisfies all 6 invariants (conservation, non-amplification, non-negativity, termination, idempotency, depth monotonicity), human judgment determines which implementation is correct.
