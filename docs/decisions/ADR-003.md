# ADR-003: Minimum Kernel Version — 4.18 vs 5.4+

- **Status:** Accepted
- **Date:** 2026-03-17

## Problem

The project needed to establish a minimum supported kernel version. This decision affects BPF program complexity (instruction limits, available helpers), stack unwinding capability, CI testing scope, and the engineering effort spent on backward compatibility. The two candidates were kernel 4.18 (RHEL 8.0 baseline) and kernel 5.4 (RHEL 8.4+ / Ubuntu 20.04 LTS baseline).

## Options Considered

### Option A: Minimum Kernel 4.18 (RHEL 8.0+)

Support all RHEL 8.x deployments from the very first release. The BPF tracepoint infrastructure needed for core probes (`sched_switch`, `sched_wakeup`, `block_rq_issue`, `block_rq_complete`, `softirq_entry/exit`) is available on 4.18.

**Pros:**
- Covers the broadest possible RHEL 8 installed base
- `bpf_get_current_task()` available since 4.8; `bpf_get_stackid(BPF_F_USER_STACK)` available since 4.6; tail calls available since 4.2 (33 max)
- All target tracepoints (`sched`, `softirq`, `block`) are available on 4.18

**Cons:**
- **4,096 instruction limit per BPF program** (raised to 1 million in 5.2). Stack delta deep unwinding must be split into extremely small tail-call programs to fit within this limit, dramatically increasing engineering complexity
- `bpf_probe_read_kernel()` unavailable; must use the older `bpf_probe_read()` which has different semantics and requires conditional compilation
- No BTF/CO-RE on vanilla 4.18 (RHEL 8.2+ backports it); non-CO-RE paths require hardcoded struct offsets that vary by exact kernel build
- Cannot mix bpf2bpf function calls with tail calls (requires 5.10); limits code organization options
- RHEL 8.0-8.1 have already reached end of full support (May 2024); RHEL 8.2-8.3 are in maintenance-only phase

### Option B: Minimum Kernel 5.4 (RHEL 8.4+ / Ubuntu 20.04 LTS)

Target the lowest kernel version that is still under active support and provides reasonable BPF capabilities.

**Pros:**
- 1 million instruction limit eliminates BPF program size concerns entirely
- `bpf_probe_read_kernel()` available, simplifying kernel memory access patterns
- BTF likely available (RHEL 8.4+, Ubuntu 20.04 with HWE kernel)
- 5.4 is the lowest still-supported LTS kernel (Ubuntu 20.04 ESM extends to 2030)
- libbpf-tools ecosystem effectively assumes 5.5+ as baseline

**Cons:**
- Excludes RHEL 8.0-8.3 installations (though these are EOL or maintenance-only)
- Some enterprise environments may still run older kernels due to change management inertia

## Analysis

The design review identified multiple Critical findings directly tied to the 4.18 instruction limit. The C-2 finding documented that the DWARF stack unwinding approach (20-layer `#pragma unroll`, each layer requiring map lookup + memory read + conditional branch + arithmetic) would conservatively require 800-1200 instructions per BPF program — approaching or exceeding the 4,096 limit even before adding event collection logic. The review noted: "The 5.2+ limit increase to 1 million instructions makes this a non-issue on newer kernels."

The C-1 finding established that ring buffer (`BPF_MAP_TYPE_RINGBUF`, introduced in 5.8) is unavailable on 4.18, requiring a complete perfarray fallback path. While the dynamic feature probing strategy (ADR-002) handles this gracefully, each such degradation path adds engineering and testing burden.

A detailed constraint analysis in the final implementation specification cataloged the 4.18 limitations:

| Constraint | 4.18 | 5.4+ | Impact |
|-----------|------|------|--------|
| BPF instruction limit | 4,096/program | 1M/program | Stack delta must use extremely small tail-call programs |
| `bpf_probe_read_kernel()` | Not available | Available | Conditional compilation required |
| BTF/CO-RE | Not on vanilla (RHEL 8.2+ backports) | Available | Non-CO-RE path needs hardcoded offsets |
| bpf2bpf + tail call mixing | Not supported (requires 5.10) | Not yet (5.4 < 5.10) | Limits code organization |

The analysis also established lifecycle facts: RHEL 8.0-8.1 ended full support in May 2024. RHEL 8.2-8.3 entered maintenance-only phase. The libbpf-tools ecosystem substantially assumes 5.5+ as its operational baseline.

The pre-execution readiness audit reinforced that the 4096 instruction limit "doubles the engineering complexity of the stack delta unwinding approach" and recommended the version floor increase.

## Decision

**Recommended minimum: 5.4** (RHEL 8.4+ / Ubuntu 20.04 LTS). **4.18 retained as best-effort** with feature-probing degradation, but no dedicated engineering resources allocated to 4.18-specific adaptation.

The rationale is threefold:
1. RHEL 8.0-8.1 are already end-of-life; RHEL 8.2-8.3 are maintenance-only
2. The 4,096 instruction limit on 4.18 makes the stack delta unwinding scheme impractically complex
3. 5.4 is the lowest kernel version that is still under active long-term support (Ubuntu 20.04 ESM to 2030)

The dynamic feature probing matrix (ADR-002) naturally supports 4.18 kernels by degrading each unavailable feature independently. On 4.18, users would see: perfarray instead of ringbuf, `#pragma unroll` bounded loops instead of `bpf_loop`, `bpf_probe_read` instead of `bpf_probe_read_kernel`, and potentially no BTF/CO-RE. The tool will still function for basic event capture and Wait-For Graph construction, but with reduced stack depth and higher overhead.

## Consequences

**Architecture:**
- The 5.4 baseline allows confident use of the 1M instruction limit, simplifying BPF program design
- Stack delta deep unwinding (Elastic-derived approach) becomes feasible without extreme tail-call fragmentation
- The feature probing infrastructure still handles 4.18 gracefully but without dedicated optimization

**Implementation:**
- CI testing prioritizes 5.4+ kernels; 4.18 testing is best-effort using virtme-ng with extracted RHEL kernel images
- Documentation explicitly states 5.4 as the recommended minimum and 4.18 as best-effort
- No conditional compilation for `bpf_probe_read` vs `bpf_probe_read_kernel` is actively maintained — the CO-RE approach handles this through `bpf_core_type_exists` checks

**Risk:**
- Some enterprise customers may push back on the 5.4 recommendation if they are locked to RHEL 8.0-8.3. The response is that the tool still works on 4.18 with degraded capability, and these RHEL versions are end-of-life
- The "best-effort" designation for 4.18 means regressions on that platform may not block releases
