# ADR-006: Graph Visualization — Layout + Rendering

- **Status:** Accepted
- **Date:** 2026-03-17

## Problem

wPerf-rs must render a Wait-For Graph as an interactive, self-contained HTML report. The graph is a directed causal structure — edges represent "thread A waits for thread B" relationships — and correct visual communication of direction, hierarchy, and bottleneck severity is essential for user diagnosis. The existing prototype (`wperf_ui_prototype.html`) used ECharts with a force-directed layout (`layout: 'force'`), which produced what the user described as "arrows flying everywhere, unclear whether the problem is the graph algorithm itself or the rendering." A replacement strategy was needed that solves the layout deficiency while preserving the prototype's already-validated interaction patterns.

## Options Considered

### Option A: Dagre layout + ECharts rendering

Use the Dagre library (a JavaScript implementation of the Sugiyama hierarchical layout algorithm) to compute strict top-to-bottom node coordinates, then feed those absolute (x, y) positions into ECharts using `layout: 'none'` mode, where ECharts handles only rendering, tooltips, zoom/pan, and click interactions.

**Pros:** Sugiyama algorithm produces deterministic, hierarchical layouts with minimized edge crossings — ideal for directed causal graphs. ECharts `graph` series with `layout: 'none'` preserves all existing tooltip HTML formatting, node click detail panels, zoom/pan, edge thickness mapping, and red/black/gray visual encoding — 100% of the prototype's validated interaction code is reusable. The code change from the prototype is minimal: change `layout: 'force'` to `layout: 'none'` and add ~15 lines of Dagre coordinate calculation. Combined size ~1.35 MB (dagre 277 KB + ECharts 1.07 MB), acceptable for an offline HTML report. Both libraries can be embedded via `include_str!` for zero external dependencies.

**Cons:** dagre.js is no longer actively maintained (though the algorithm is stable and the graph scale of 10-50 nodes is well within its capacity). Total bundle size is larger than alternatives. Dagre alone does not solve the edge density problem — the prototype data showed 42 edges across 13 nodes (density 3.23 edges/node) with 67% noise edges below 50ms, requiring visual encoding (gray dashed lines, low opacity) as a co-solution.

### Option B: Cytoscape.js with cytoscape-dagre plugin

Use Cytoscape.js as both the layout engine and rendering engine. The cytoscape-dagre plugin provides the same Sugiyama hierarchical layout. Cytoscape handles all rendering and interaction.

**Pros:** Smaller bundle (~396 KB minimal). Dagre layout is a one-line configuration (`layout: { name: 'dagre', rankDir: 'TB' }`). Actively maintained.

**Cons:** Cytoscape lacks built-in HTML-formatted tooltips — requires additional libraries (popper.js/tippy.js) for custom tooltip implementation. Zero code reuse from the existing prototype: all ECharts-based interaction code (tooltip formatters, click handlers, visual encoding logic) would need to be rewritten from scratch against the Cytoscape API. The final tooltip and interaction quality may not reach the level already validated with users in the ECharts prototype. This represents a significant implementation risk with no corresponding benefit.

### Option C: Rust-side Dagre (petgraph) + ECharts rendering (no dagre.js)

Compute the Sugiyama hierarchical layout in the Rust `wperf report` backend using petgraph's topological sort and a custom or ported Sugiyama implementation. Emit pre-computed (x, y) coordinates in the JSON data. The HTML report uses only ECharts for rendering (no dagre.js dependency).

**Pros:** Eliminates the 277 KB dagre.js frontend dependency. Aligns with the "backend computes, frontend has zero logic" design philosophy. Same ECharts rendering and interaction as Option A.

**Cons:** petgraph provides topological sort but has no Sugiyama hierarchical layout implementation. Porting or implementing Sugiyama in Rust is significant engineering effort with uncertain maturity. This is a premature optimization — the 277 KB saving is negligible in a self-contained HTML file that already embeds ~1 MB of ECharts.

## Analysis

The visualization review traced the "arrows flying everywhere" problem through its full diagnostic chain. The user originally posed two possible causes: the graph algorithm itself, or the rendering. The first design review diagnosed this as a single cause — the force-directed layout's inherent properties (random initialization, non-deterministic positions, no directional hierarchy). This diagnosis was accepted without objection in the reply, and codified in the final implementation spec.

A subsequent dedicated visualization review (the 22nd document in the spec chain) performed a multi-factor decomposition using quantitative data from the actual prototype. The prototype contained 13 nodes, 42 edges (density 3.23 edges/node — very dense), 28 noise edges below 50ms (67% of all edges), and 2 self-loops (Handler at 7540ms, ParallelGC at 2491ms). The analysis concluded that "arrows flying everywhere" was a multi-factor problem:

1. **Force layout's non-determinism and lack of direction** (the primary factor, correctly identified by the original review)
2. **Extreme edge density** (42/13 = 3.23, causing edge crossings even in hierarchical layouts)
3. **Visual noise from 28 low-weight edges** (67% of edges carry negligible information)
4. **Self-loops** (any layout requires special handling)

The visualization review confirmed that Dagre addresses factors 1 (deterministic, hierarchical, directed) but factors 2-4 require the visual encoding system (gray dashed lines for noise, red/black/gray color coding by weight threshold, opacity management) as a co-solution. The conclusion was explicit: Dagre layout + visual encoding is a "joint solution — neither alone is sufficient."

The review also verified that all existing ECharts interaction patterns (HTML tooltip formatting, node click detail panels, zoom/pan, edge width proportional to `attributed_delay_ms`, Knot SCC red dashed border marking) are bound to ECharts' rendering layer and are completely orthogonal to the layout algorithm. Switching from `layout: 'force'` to `layout: 'none'` preserves all of them.

Cytoscape.js was evaluated and rejected primarily because it would require rewriting all validated interaction code with no improvement in layout quality (both use the same underlying Dagre/Sugiyama algorithm). The Rust-side Dagre option was deferred to a future optimization phase (Phase 3+) since the engineering effort is disproportionate to the 277 KB saving.

## Decision

Adopt Option A: Dagre (Sugiyama) for layout computation + ECharts with `layout: 'none'` for rendering and interaction. This is a division-of-labor architecture: Dagre calculates strict top-to-bottom DAG coordinates, ECharts renders and provides all interactivity.

## Consequences

- **Layout quality:** Deterministic hierarchical layout with minimized edge crossings. Nodes flow top-to-bottom representing causal direction (waiter above, waitee below). Eliminates the random, non-directional nature of force-directed layout.
- **Prototype reuse:** Tooltip formatters, click handlers, visual encoding, zoom/pan — all existing code is preserved. Migration requires changing one parameter (`layout: 'force'` to `layout: 'none'`) and adding ~15 lines of Dagre coordinate calculation.
- **Bundle size:** ~1.35 MB total (embedded via `include_str!`), acceptable for offline HTML reports.
- **Incomplete solution:** Dagre alone does not solve the edge density problem. The visual encoding system (noise edge filtering/fading, weight-proportional thickness, color coding) must be implemented concurrently — delivering "bare Dagre" without visual encoding will reproduce the visual clutter from edge density.
- **Maintenance risk:** dagre.js is not actively maintained. For 10-50 node graphs this is not a functional concern, but long-term the Rust-side Sugiyama option (Option C) should be evaluated as a Phase 3+ optimization.
- **Self-loops:** Require special rendering treatment (loop-back arrows) in ECharts, as hierarchical layout does not naturally handle them.
