# ADR-004: Event Transport — Ring Buffer Strategy

- **Status:** Accepted
- **Date:** 2026-03-17

## Problem

The BPF-to-userspace event transport is the data backbone of wPerf-rs. Every `sched_switch`, `sched_wakeup`, `block_rq`, and `softirq` event must travel from kernel BPF programs to the Rust user-space collector. The transport choice affects data ordering guarantees, per-CPU overhead, kernel version compatibility, and the maximum sustainable event rate before packet loss.

Ring buffer (`BPF_MAP_TYPE_RINGBUF`, introduced in Linux 5.8) is the modern preferred transport, but the project targets kernels as old as 4.18 (where ringbuf does not exist). The project needed a strategy that provides optimal performance on modern kernels while remaining functional on older ones.

## Options Considered

### Option A: Ring Buffer Only

Declare `BPF_MAP_TYPE_RINGBUF` as the sole transport. Set the minimum supported kernel to 5.8+.

**Pros:**
- Single code path, simplest implementation
- Global ordering of events (ringbuf's spinlock-serialized reservation ensures consumption order matches reservation order, regardless of producer CPU)
- Shared memory pool across all CPUs eliminates per-CPU buffer waste (no False Sharing)
- Backpressure support via `BPF_RB_FORCE_WAKEUP` and related flags

**Cons:**
- Completely excludes kernels before 5.8, including RHEL 8 through 8.3 (4.18-based) and Ubuntu 18.04 LTS (4.15/5.4)
- BPF program load fails with `EINVAL` on <5.8 kernels — not a degradation but a hard failure

### Option B: Perf Event Array Only

Use `BPF_MAP_TYPE_PERF_EVENT_ARRAY` exclusively. Available since kernel 4.3, ensuring universal compatibility.

**Pros:**
- Works on all target kernels from 4.18 upward
- Well-understood, mature transport mechanism

**Cons:**
- Per-CPU buffers: each CPU has an independent ring, requiring user-space to `epoll` multiple file descriptors
- Events from different CPUs arrive in arbitrary order; user-space must perform timestamp-based reordering
- No backpressure: when a per-CPU buffer fills, events are silently dropped (counter incremented)
- Memory waste: each per-CPU buffer must be sized for peak load, but most CPUs may be idle, wasting allocated memory
- Higher overhead at high event rates due to per-CPU lock contention and cross-CPU memory access patterns

### Option C: Single-ELF CO-RE Dual-Mode with Auto-Detection

Compile a single BPF object file that declares both a ringbuf map and a percpu-array staging map. At BPF load time, use `bpf_core_type_exists(struct bpf_ringbuf)` to select the active code path. In user-space, probe ringbuf availability before loading, and reconfigure map types accordingly.

**Pros:**
- Single `.bpf.o` file: one skeleton, one build artifact, one Rust loading path
- Optimal transport on 5.8+ kernels (ringbuf with global ordering and shared memory)
- Functional transport on 4.18-5.7 kernels (perfarray with user-space reordering)
- Follows the proven `compat.bpf.h` pattern from bcc/libbpf-tools, validated in 5+ production tools
- User-space `EventTransport` trait abstracts consumption differences cleanly

**Cons:**
- BPF source contains both paths, increasing code complexity
- Perfarray path requires a Min-Heap Reorder Buffer in user-space (adds ~100 lines and 50ms latency window)
- Two transport behaviors means two testing configurations
- The `heap` percpu-array (used as staging area for `bpf_perf_event_output`) adds a map that is unused on ringbuf kernels

## Analysis

The design review (C-1 finding) was the catalyst for this decision. The original design assumed ringbuf universally while simultaneously claiming 4.18 support. The review stated: "`BPF_MAP_TYPE_RINGBUF` was introduced in Linux 5.8 (commit `457f4436`). The 4.18 kernel has no such Map type. This means there is a fundamental contradiction between the documented minimum kernel (4.18) and the core data transport mechanism."

The design team's initial response proposed an `EventTransport` trait with separate `RingBufTransport` and `PerfEventArrayTransport` implementations. However, the secondary review identified a critical implementation gap: the two transports require different BPF map type declarations at compile time, meaning "two separate `.bpf.o` files must be compiled, with the user-space loader selecting which to load." This would produce "two skeleton files, two build artifacts, `#define` isolation prone to missed changes, and two Rust-side loading paths."

Research into the bcc/libbpf-tools codebase revealed the `compat.bpf.h` pattern, already validated in production tools (filelife, opensnoop, oomkill, mountsnoop, tcppktlat). This pattern uses a single ELF file with both map types declared, a `reserve_buf()`/`submit_buf()` abstraction layer in BPF that uses `bpf_core_type_exists(struct bpf_ringbuf)` for runtime path selection, and user-space map reconfiguration before load.

The specific mechanism works as follows:
- BPF-side: `reserve_buf()` calls `bpf_ringbuf_reserve()` if `bpf_core_type_exists(struct bpf_ringbuf)` is true, otherwise falls back to `bpf_map_lookup_elem(&heap, &zero)` to get a percpu staging slot
- BPF-side: `submit_buf()` calls `bpf_ringbuf_submit()` or `bpf_perf_event_output()` accordingly
- User-side: `probe_ringbuf()` attempts `bpf_map_create(BPF_MAP_TYPE_RINGBUF)` as a capability check. If successful, `heap.set_autocreate(false)` suppresses the unused staging map. If ringbuf is unavailable, the `events` map is reconfigured to `PerfEventArray` type before loading

The perfarray path introduces cross-CPU event ordering challenges. The `bpf_ktime_get_ns()` timestamp, while based on `ktime_get_mono_fast_ns()` which is not strictly guaranteed monotonic across CPUs during NTP slope updates, is practically monotonic on modern x86 hardware with `constant_tsc` + `nonstop_tsc`. The 50us false-wakeup pruning threshold already filters events in the time range where timestamp inversions could occur.

The final specification codifies the libbpf `probe_ringbuf()` pattern: user-space attempts `bpf(BPF_MAP_CREATE)` with `BPF_MAP_TYPE_RINGBUF`; success means ringbuf is available; failure triggers perfarray reconfiguration.

## Decision

**Option C: Single-ELF CO-RE dual-mode with auto-detection** was chosen.

A single `wperf.bpf.o` contains both ringbuf and percpu-array map declarations with a `reserve_buf()`/`submit_buf()` BPF-side abstraction. User-space probes ringbuf availability at startup using `probe_ringbuf()` and reconfigures maps accordingly before loading:

- **Ringbuf available (5.8+):** `heap` map autocreation is disabled; events flow through ringbuf with global ordering guarantee
- **Ringbuf unavailable (<5.8):** `events` map is reconfigured to `PerfEventArray`; user-space Min-Heap Reorder Buffer (50ms window) restores global ordering by timestamp

## Consequences

**Architecture:**
- Single BPF compilation target eliminates divergent build paths. The `compat.bpf.h` pattern keeps both transport paths in one source file with CO-RE conditionals
- User-space `EventTransport` trait provides a unified interface: `fn poll(&mut self) -> Vec<WperfEvent>`, hiding the ringbuf vs perfarray distinction from the collector and analysis pipeline
- The Min-Heap Reorder Buffer in the perfarray path adds 50ms of latency between event capture and analysis, acceptable for an offline profiling tool

**Implementation:**
- The `heap` percpu-array map (declared but autocreate-disabled on ringbuf kernels) adds one map declaration to the BPF source with zero runtime cost on modern kernels
- Perfarray path requires `epoll` over N file descriptors (one per CPU), adding user-space polling complexity
- Event reordering buffer must handle the case where `bpf_ktime_get_ns()` produces non-monotonic timestamps across CPUs during NTP slope updates; the state machine uses best-effort association for <1us out-of-order events rather than panicking
- The 50ms reorder window means that events delayed beyond this threshold are processed out of order; this window is tuned to exceed the maximum expected cross-CPU timestamp skew by orders of magnitude

**Risk:**
- The perfarray path has no backpressure mechanism: when a per-CPU buffer fills, events are silently dropped. The `drop_counter` in BPF code must be exposed as a user-visible metric during `wperf record` so operators can increase buffer sizes or reduce collection scope
- Two transport behaviors require integration testing on both ringbuf-capable and ringbuf-incapable kernels. The virtme-ng cross-kernel testing infrastructure must include at least one kernel below 5.8 to exercise the perfarray path
- The CO-RE conditional `bpf_core_type_exists(struct bpf_ringbuf)` is resolved at BPF load time by the libbpf relocator; if libbpf itself has bugs in this resolution (unlikely given production usage), both paths could fail simultaneously
