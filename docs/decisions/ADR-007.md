# ADR-007: Cascade Algorithm Verification Strategy

- **Status:** Accepted
- **Date:** 2026-03-17

## Problem

The Cascade Engine (level-redistribution algorithm) is wPerf's only custom core algorithm — there is no third-party production-grade implementation anywhere in the world. A search of GitHub and major open-source projects (lockdep, PostgreSQL deadlock.c, Perfetto, HPCToolkit, Parca, Pyroscope, SchedViz) confirmed the only existing implementation is the original wPerf Python analyzer (~51 stars, research prototype quality, NetworkX dependency). The algorithm recursively propagates wait-time weights along directed edges using time-window overlaps and proportional allocation — significantly more complex than boolean deadlock detection (which PostgreSQL has refined over 25 years). During the pseudocode review process alone, 5 algorithmic bugs were discovered across 3 review iterations:

| Bug | Severity | Nature |
|-----|----------|--------|
| BUG-1 | P0 | `visited_path` insert/remove placed inside the loop body instead of outside, causing inter-path contamination of the cycle-detection set |
| BUG-2 | P0 | `propagated_down` return value discarded, making transfer penalties independent of downstream absorption |
| BUG-3 | P1 | Multiple outgoing edges with overlapping time windows cause double-counting of the same time interval |
| BUG-4 | P1 | Combined effect of BUG-2 and BUG-3 breaks weight conservation (attributed totals diverge from raw totals) |
| NEW-BUG-1 | P0 | `child_self_blame` discarded from subtree absorption calculation, causing all leaf nodes (including pseudo-threads like Network/Disk) to receive zero attribution |

The archived project's history provides concrete evidence of the risk: `cascade.rs` was modified 44 times (on average every 34 lines of code was changed once), and the overall project had a 42.8% fix-to-commit ratio with the last 3 days showing 64% fix commits. The algorithm is a proven high-error-surface area.

## Options Considered

### Option A: Unit tests only

Write a conventional test suite with hardcoded input graphs and expected outputs. Rely on human reasoning to select representative test cases.

**Pros:** Familiar, fast to write, deterministic.

**Cons:** Human-selected test cases are biased toward anticipated failure modes. The 5 discovered bugs were all found by manual review, not by testing — suggesting that standard unit tests would have missed them. Cannot guarantee coverage of the combinatorial space of graph topologies, time-window overlaps, concurrent waiters, and recursion depths. Provides no structural guarantee that the test suite itself is adequate.

### Option B: Property-based testing only

Use `proptest` to generate random graphs and verify mathematical invariants hold.

**Pros:** Explores far more graph topologies than hand-written tests. Can verify universal properties (conservation, non-negativity, non-amplification, termination) across thousands of random inputs.

**Cons:** Random generation may not target the specific pathological configurations that trigger known bugs (e.g., the exact insert/remove scope of BUG-1 requires a specific multi-outgoing-edge topology). Provides no regression guarantee for known bugs. Cannot validate against a reference implementation. Offers no measure of test suite quality.

### Option C: Five-layer verification pyramid

A layered strategy combining complementary verification techniques:

1. **Layer 1 — Invariant assertions (Day 1):** Embed `debug_assert!` checks for 6 mathematical invariants (I-1 weight conservation, I-2 non-amplification, I-3 non-negativity, I-4 termination, I-5 idempotency, I-6 depth monotonicity) directly in the algorithm. The JSON output's `is_conserved` field serves as the final production-level sentinel.

2. **Layer 2 — Paper scenarios + regression tests (Week 1):** Hardcoded tests for the OSDI'18 paper's Figure 4 scenario (User-Parser-Network linear chain) and a dedicated regression test for each of the 5 known bugs, ensuring they never regress.

3. **Layer 3 — Property-based testing (Week 2):** `proptest`-driven random graph generation (10K+ graphs, biased toward realistic characteristics: sparse, mostly acyclic, depth 3-8) verifying all 6 invariants hold universally.

4. **Layer 4 — Differential testing vs. Python reference (Week 2-3):** Run the same input graph through both the Rust implementation and the original wPerf Python analyzer (`bottleneck.py`), comparing per-edge `attributed_delay` values within a 1.0ms tolerance.

5. **Layer 5 — Mutation testing (Week 3):** Use `cargo-mutants` to inject syntactic mutations (delete `path.insert`, flip arithmetic operators, change comparison boundaries) and verify the test suite detects at least 95% of mutations.

**Pros:** Each layer addresses a different class of risk. Invariant assertions catch any conservation-breaking bug at runtime (I-1 alone catches 4/5 known bugs). Paper scenarios validate semantic correctness against the authoritative source. Property testing covers the combinatorial explosion human reasoning cannot. Differential testing catches conceptual misunderstandings of the algorithm. Mutation testing validates the test suite itself. The layers are complementary — no single layer catches everything, but together they provide defense in depth.

**Cons:** Significant upfront investment (estimated 3 weeks for full implementation). The Python reference may itself contain bugs (acknowledged in the specification — if Rust and Python disagree but Rust satisfies I-1 through I-6, human judgment is needed). Mutation testing tooling maturity for Rust is still evolving.

## Analysis

The pseudocode review process provided the strongest evidence for the need for multi-layer verification. Five bugs were discovered through human review of approximately 100 lines of pseudocode — an error density of roughly 1 bug per 20 lines. Moreover, the bug fix process itself introduced a new bug: the BUG-2 fix (incorporating `propagated_down` into the transfer penalty calculation) initially used only the downstream propagation amount, discarding the child node's own blame, which caused all leaf nodes (the most common attribution targets in real graphs) to receive zero weight.

Analysis of the 6 proposed invariants against the known bug corpus revealed that weight conservation (I-1) is the single most powerful detector: it directly catches BUG-2, BUG-3, BUG-4, and NEW-BUG-1 (4 out of 5 known bugs). BUG-1 (visited_path scope error) is only partially caught by I-4 (termination) — it causes incorrect but finite traversals, not infinite recursion. The pre-execution readiness audit proposed a 7th invariant, I-7 (locality: weight flows only along existing edges and within time-window intersections), which would specifically catch BUG-1-class path errors.

The archived project's failure provides a cautionary reference: `cascade.rs` accumulated 44 modifications, and spec assertions were relaxed 9 times in 7 days to match incorrect output — meaning the test infrastructure itself became untrustworthy. The three-prohibition policy (no modifying specs to match output, no workarounds bypassing earlier phases, no concurrent multi-phase development) directly addresses this failure mode.

The comparison with industry practice reinforces the approach: AWS DynamoDB uses TLA+ formal verification for distributed consistency (discovering 35-step error paths that testing could not cover); CockroachDB combines TLA+ with property-based testing; PostgreSQL's deadlock detection has been refined over 25 years but solves a fundamentally simpler problem (boolean cycle detection vs. recursive weighted propagation).

## Decision

Adopt Option C: the five-layer verification pyramid. Weight conservation (I-1) must be implemented on Day 1 as it catches 4 out of 5 known bugs with approximately 20 lines of assertion code. Each of the 5 known bugs must have a dedicated regression test before any further algorithm development. Property-based testing must exercise 10,000+ random graphs with zero invariant violations. Differential testing against the Python reference validates conceptual correctness. Mutation testing validates test suite quality with a target detection rate of 95% or higher.

## Consequences

- **Development timeline:** Phase 0 is extended from 2 weeks to 3 weeks to accommodate the full verification pyramid. This is a deliberate trade: the archived project spent 9 days in a fix spiral (66 commits, 42 fixes) that could have been prevented by upfront verification.
- **Invariant I-1 as sentinel:** `is_conserved` in the JSON output is the production-level final defense. Any conservation violation in CI or production is a hard failure that blocks release.
- **Python oracle limitation:** The differential testing layer depends on a research-prototype-quality reference. Disagreements require human judgment. Both implementations may share conceptual errors (common-mode failure), which the invariant and property layers help catch independently.
- **Ongoing cost:** Mutation testing and property testing add CI time. The recommended configuration runs 10K property tests in standard CI and 100K in nightly builds.
- **Design-for-testability:** The cascade engine must be a pure function (immutable graph input + parameters, new graph output, no side effects). Hash-map iteration order must be deterministic (`BTreeMap` or sorted keys). A `cascade_engine_with_trace()` variant enables debugging intermediate recursive decisions.
- **I-7 (locality) as recommended addition:** A 7th invariant asserting that weight flows only along existing edges and within time-window intersections would specifically address BUG-1-class errors that I-1 does not fully catch.
