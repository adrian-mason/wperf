# ADR-010: Binary Data Format — wPRF v1

- **Status:** Accepted
- **Date:** 2026-03-17

## Problem

wPerf separates data collection (`wperf record`) from analysis (`wperf report`) as independent CLI commands. The intermediate binary file (`.wperf`) is the data contract between these two components. Without a rigorous format specification, the two components cannot be developed in parallel, version mismatches will silently corrupt data, and collection interruptions (SIGKILL, power loss) will render entire capture files unreadable.

The archived project provides a cautionary precedent: the binary format was changed 5 times (wPRF1 through wPRF5, with evidence of a commit bumping the magic from wPRF4 to wPRF5) with no forward compatibility mechanism. Each format change required coordinated updates across writer and reader code, and old capture files became unreadable.

## Options Considered

### Option A: Evolving format (wPRF1-5 as in archived project)

Continue the pattern from the archived project: a version byte in the header, with the reader rejecting files whose version does not match the expected constant. Format changes increment the version, and old files are abandoned.

**Pros:** Simplest implementation — no need for forward compatibility logic. Writer and reader are always exactly synchronized.

**Cons:** The archived project's 5 format changes in a single development cycle demonstrate this approach's fragility. Every schema change breaks all existing capture files. Parallel development of writer and reader is impossible without constant coordination. No crash tolerance — if the writer is interrupted, the file has no recovery mechanism. No extensibility — adding a new metadata section or event type requires a version bump that invalidates all existing tooling.

### Option B: Clean v1 design with forward compatibility (TLV + version byte + feature bitmap)

Design a single durable v1 format incorporating lessons from the archived project's failures and from mature binary formats (notably `perf.data`). Key design elements:

1. **64-byte fixed header** containing magic ("wPRF"), version (1), endianness, architecture, a `data_section_end_offset` for crash recovery, a `section_table_offset` for footer location, and a 256-bit feature bitmap.

2. **TLV (Type-Length-Value) record wrapping**: Every event in the data stream is prefixed with a 5-byte `RecordHeader` (1-byte type + 4-byte length). Parsers encountering an unknown record type can skip `length` bytes to continue reading, enabling forward compatibility without version bumps.

3. **Footer Section Table**: Metadata sections (String Table, Symbol Resolution Table, Build-ID, hostname/OS release/CLI args) are stored at the end of the file, referenced by `(Section_ID, Offset, Size)` tuples. New sections can be added without changing the header or event stream format.

4. **Crash tolerance**: The `data_section_end_offset` field in the header is updated periodically during capture. If the footer is lost (SIGKILL, power failure), the parser can still scan forward through the event stream up to this offset, recovering all complete events. This exceeds `perf.data`'s crash tolerance.

5. **256-bit feature bitmap**: Describes compression type, supported event types, probe capabilities, etc. Replaces the original 1-byte `meta_flags` field (only 8 bits) identified as insufficient in review.

6. **DoS protection**: `PayloadChunk` variable-length records enforce a maximum payload size (16MB) to prevent memory exhaustion from malicious or corrupted files.

**Pros:** Single format version supports incremental extension. TLV allows parsers to skip unknown record types gracefully. Feature bitmap enables capability negotiation. Crash tolerance makes partial captures recoverable. Footer Section Table supports arbitrary metadata without schema changes. Validated by Rust implementation (correct 64-byte layout, symmetric read/write, version check, DoS protection).

**Cons:** More complex initial implementation than a bare version-bump format. The feature bitmap and section table add bytes to every file even when unused. The footer-at-end design means metadata is only available after a complete read (though the crash tolerance offset enables streaming event access).

## Analysis

The need for a binary format specification was first identified as a critical gap (designated N-4) during the second expert review: the architecture split `record` and `report` into independent binaries but left the "data umbilical cord" undefined, making parallel development impossible.

The initial format proposal (wPRF4, a 32-byte header) was found to have five critical deficiencies in the third expert review:

1. **No Build-ID section** — binary upgrades would cause symbol resolution misalignment on old capture files
2. **Feature bits limited to 1 byte (8 bits)** — insufficient expansion space (perf.data uses 256 bits)
3. **No Attr section** — collection parameters (which events, sampling rate) could not be recovered from the file
4. **No host metadata** — cross-machine analysis could not distinguish file origins
5. **No crash tolerance** — collection interruption made the entire file unreadable

Additionally, the header claimed 32 bytes but only accounted for 24 bytes of fields, with 8 bytes unreserved. The compression scheme was internally contradictory (body text said "GZipped/Snappy" while the narrative claimed Zstd). The claimed 20:1 compression ratio was found to be unrealistic for scheduling event data with high-entropy nanosecond timestamps (actual Zstd ratio is 5:1 to 8:1 based on perf.data empirical data of 3:1 to 7:1).

The Rust implementation (wPRF4 code artifact) was reviewed and found to have a correct 64-byte layout (4+1+1+1+1+8+8+32+8 = 64, verified byte-by-byte), symmetric read/write field ordering, and proper version validation. A DoS vulnerability in `PayloadChunk` (unbounded allocation from `u32` length) was identified and fixed with a 16MB maximum payload check. The TLV `RecordHeader` (type + length = 5 bytes) was added to enable skipping unknown record types, directly addressing forward compatibility.

The archived project's format evolution (5 versions with no forward compatibility) validated the decision to invest in a durable v1 design. Each archived format change represented wasted engineering effort and abandoned capture files — exactly the failure mode that TLV wrapping and the section table are designed to prevent.

## Decision

Adopt Option B: A clean wPRF v1 format with forward compatibility through TLV record wrapping, a 256-bit feature bitmap, footer section tables, and crash tolerance via `data_section_end_offset`.

## Consequences

- **Forward compatibility:** New record types and metadata sections can be added without bumping the format version. Older parsers skip unknown TLV records and unknown section IDs gracefully. This eliminates the format churn that plagued the archived project.
- **Crash tolerance:** The `data_section_end_offset` field enables recovery of all complete events even when the footer is lost. This exceeds `perf.data`'s crash recovery capabilities and is critical for production use where SIGKILL and power loss are real scenarios.
- **Parallel development:** The format specification serves as a stable contract between `wperf record` and `wperf report` development tracks. Changes to one side do not require lockstep changes to the other as long as TLV discipline is maintained.
- **File overhead:** The 64-byte header, 5-byte TLV headers per record, and footer section table add modest overhead. For a typical capture file with millions of events, the per-event 5-byte TLV header is the dominant addition (~5 bytes / ~28 bytes per BaseEvent = ~18% overhead), which is an acceptable trade for forward compatibility.
- **Compression planning:** Capacity planning must use the corrected 5:1 to 8:1 Zstd compression ratio, not the original 20:1 estimate. At 100K events/s with ~28 bytes per event, raw throughput is ~2.8 MB/s, compressed to ~350-560 KB/s.
- **Remaining specification gaps:** The Attr section (recording collection configuration), the complete byte-level header RFC, and the explicit parser behavior for unknown Section IDs are deferred to implementation phase (Phase 1 Week 1). These are P2 items that do not block the core format design.
- **Security:** The 16MB maximum payload size on `PayloadChunk` prevents memory exhaustion from malicious or corrupted files, a defense-in-depth measure identified and implemented during the code review cycle.
